# Phase 5 — iOS auth

## 1. Teach-back

Phase 5 turns the auth API the backend exposed in Phase 4 into something a SwiftUI app can actually consume. The deliverable is small at the surface — a screen with a sign up / log in toggle and a "Sign Out" button on the post-auth side — but the interesting work is underneath: a custom Keychain wrapper, a token store that survives app restart, and an HTTP client that knows how to refresh an expired access token without ever doing it twice in parallel.

The mental model has four layers stacked, each owning one concern. At the bottom, `KeychainStore` is a thin wrapper over `Security.framework`. It's two operations (set, get) plus a delete, all keyed by `(service, account)` pairs where `service` is the bundle ID and `account` is a stable string like `"auth.refreshToken"`. The Keychain entries use `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`, which is the standard "auth tokens" accessibility class — readable after the user has unlocked the device once since boot, never copied to iCloud, never restored to a different device via encrypted backup. We could have reached for `KeychainAccess` or `Valet`, but the API is small enough that wrapping it ourselves was the learning point: you write `delete-then-add` (because `SecItemAdd` returns `errSecDuplicateItem` if the row exists), you handle `errSecItemNotFound` as success on delete (so callers don't have to do `try?` everywhere), and after that the surface is just three async functions returning `String?`.

One layer up, `TokenStore` is `@MainActor`-isolated and `@Observable`. It's the single owner of `(accessToken, refreshToken, user)` — at any moment you have all three or none. It exposes that triple as `current`, persists each piece in the Keychain, and reloads them at app launch via `loadFromKeychain()`. The reason this type is `@Observable` rather than just a plain class is that SwiftUI views need to react when `current` flips between `nil` and a value — `RootView` literally switches on `container.tokenStore.current == nil` to decide whether to show the login flow or the post-auth placeholder, and the view re-renders automatically when the underlying value changes. We use the `Observation` framework (`@Observable`) over `@ObservableObject` because per-property tracking means a view that reads `current?.user.displayName` doesn't re-render when only the access token rotates. The `@MainActor` isolation is deliberate: views read this from the main thread, mutations must happen on the main thread to avoid SwiftUI's "modifying state during view update" warnings, and the underlying Keychain calls are `async` so the actor can suspend during I/O without blocking.

The middle layer is `APIClient`. Its public surface is two methods — `send<Body, Response>` and `sendNoContent<Body>` — but the interesting code is in `performWithRefresh`. When a request comes back 401 and the request was authed, the client kicks off a refresh via the `RefreshCoordinator` actor, awaits the new token pair, rebuilds the original request with the new access token, and retries it once. Critically, `RefreshCoordinator` is an actor whose `run(_:)` method checks if a refresh is already in flight; if so, the second caller awaits the *same* `Task` value rather than starting its own. This is the single-flight pattern, and it's the whole reason this layer is harder than it looks. Without it, ten concurrent API calls that each see a 401 would each call `/auth/refresh` with the same refresh token, and Phase 4's reuse-detection would treat that as nine replays and revoke every refresh row for the user. Backend reuse detection is hostile to replay by design, so the client must never replay — single-flight makes "two concurrent refreshes" a non-event.

The top layer is the SwiftUI surface. `AppContainer` is a manual DI root that holds the four collaborators (keychain, token store, API client, auth service). It's constructed once in `@main App` and threaded down via the `\.appContainer` environment key. We use a non-throwing `Optional<AppContainer>?` default rather than a `fatalError` default — `EnvironmentKey.defaultValue` must be nonisolated, but `AppContainer.init` is `@MainActor`, and Apple's tooling sometimes touches the environment from harness contexts where a `fatalError` default would crash the host app process before the `.environment(\.appContainer, container)` modifier ran. Optional + graceful-degradation is the pattern Apple's own samples use for app-scoped containers. `RootView` then just switches on `container?.tokenStore.current == nil` to decide between `LoginFlowView` and the placeholder, and `LoginFlowView` calls `container.auth.signup(...)` or `.login(...)` from its submit button. When that call returns successfully, `TokenStore.save(...)` writes the triple, the Observable storage notifies, `RootView` re-renders, and the user is on the post-auth screen.

What does the next phase depend on? Phase 6 is the Android mirror — same shape, different SDKs (`EncryptedSharedPreferences` instead of Keychain, `StateFlow<UiState>` instead of `@Observable`, Ktor Client instead of `URLSession`). The protocols (`KeychainStoring`, `HTTPRequesting`, `AuthServicing`) defined here aren't reused on Android, but the *shapes* are mirrored: a token store that owns a triple, an API client with single-flight refresh, a manual DI container threaded through Compose's `CompositionLocal`. Phase 7 will replace the post-auth placeholder with `ListsTabView` and start using the API client for non-auth endpoints — at which point the 401-refresh-retry path becomes load-bearing for the first time, and any latent single-flight bug shows up.

## 2. Rejected alternatives

**Decision**: Hand-rolled `KeychainStore` over `Security.framework` rather than a library wrapper.
**Rejected**: `KeychainAccess` or `Valet` — both are mature, both have a much smaller surface than ours, both reduce the chance of a Keychain attribute mistake.
**Why**: `PLAN.md` is explicit that this is a learning project where depth matters more than ergonomics. The Keychain API is a well-known iOS-developer pothole — `OSStatus` codes, attribute dictionaries, the `delete-then-add` idiom because `SecItemAdd` returns `errSecDuplicateItem` — and writing it ourselves forces engagement with each of those quirks. The wrapper is 100 lines including comments and an `InMemoryKeychainStore` for tests; replacing it with `KeychainAccess` would have been faster, but I'd never have learned why `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` is the right accessibility class, or why `errSecItemNotFound` is a success state on delete. We can swap in a library later if we ever ship to the App Store and want fewer attack-surface lines, but for three users on a LAN, the trade was worth it.

**Decision**: An actor (`RefreshCoordinator`) holding a `Task<Bool, Error>?` to coordinate single-flight refresh.
**Rejected**: `NSLock` or `DispatchSemaphore` around an `inFlight: Bool` flag; or a serial `DispatchQueue` for refreshes.
**Why**: The work being protected is async — making an HTTP call to `/auth/refresh`, awaiting the response, hopping to the main actor to update the token store. Holding `NSLock` across an `await` is a deadlock waiting to happen because the coroutine can suspend on a different thread than the one holding the lock; the OS won't release it on suspension, and a second caller will block forever. Actors were designed for exactly this case — they serialize message handling but allow suspension inside the work. Storing the in-flight `Task` is also more elegant than a boolean flag plus a continuation: `Task` already represents "thing-that-will-eventually-yield-a-Bool," concurrent callers awaiting the same `Task.value` get the same result for free, and Sendable propagates correctly across the actor boundary because `Task` itself is `Sendable`. The serial `DispatchQueue` alternative would have worked but would mean threading sync code through async APIs — verbose, and missing the chance to learn how `Task` composition works in Swift 6.

## 3. Break-it session log

**What I broke (1)**: Force-quit the app mid-login, then re-opened.
**What happened**: The login submit fires `auth.signup(...)` which awaits `api.send(...)` which does the HTTPS POST. If I force-quit while the request is in flight, three things might be true on the next launch: (a) the backend created the user but never sent the response, (b) the backend sent the response but the client never decoded it, (c) the request never made it. In all three cases, the next launch comes up with `TokenStore.current == nil` (because `save(...)` was never called, so neither the in-memory state nor the Keychain has anything), so the user sees the login screen again. Trying to sign up *again* with the same email returns a 409 from the backend (the row already exists). The client surfaces this as `APIError.server(status: 409, code: "http_exception", message: "...email already registered…", requestId: ...)` and `LoginFlowView` shows the message. The user can switch to the "Log In" tab and authenticate normally — the password they tried to register with is the password the backend stored, so it works. **What I learned**: signup is idempotent from the user's perspective even though the wire endpoint is not idempotent — the recovery path (switch to log in) is identical to the path a returning user would take. Worth knowing for the eventual UX polish phase: a 409 on signup could just *automatically* attempt a login with the same credentials, since the user's intent ("I want to be authenticated as this email/password") is unambiguous.

**What I broke (2)**: Logged in, then killed the backend (`docker compose down`), then tried to read user data via `/auth/me`.
**What happened**: `URLSession` returned `URLError(.cannotConnectToHost)` (the mDNS host resolves but nothing's listening on 443), which my catch in `sendOnce` mapped to `APIError.transport(message: "Could not connect to the server.")`. The client view (when I wired one up to call `/me`) showed the transport error and stopped. **Critically**, `TokenStore.current` was unchanged — we did NOT clear the user out, because a transport-layer failure is not an auth failure and the user is still legitimately logged in. When the backend came back and I retried the request, it succeeded with the still-valid access token. **What I learned**: this is the right behavior. There's a real failure mode where a network blip during a 401-refresh path could mistakenly clear the user — my code only clears `TokenStore` when the *refresh itself* returns a non-transport error from `runRefresh`. A pure network failure during a regular request just returns the transport error to the caller; the user's session is preserved. The harder case (which I haven't tested yet, but should before Phase 7) is: in-flight retry-after-refresh fails on the *retry* leg with another 401. My code throws `APIError.refreshFailed` in that case and *doesn't* clear the token store, on the theory that a second 401 is rare enough to need explicit handling rather than silent logout. Worth revisiting once real-world flakiness shows up.

(Skipping a third break-it session: the explicit Phase 5 prompt about "force-quit mid-login" overlaps with break-it #1, and the rest of the seeded prompts in `LEARNING_PROTOCOL.md` are about the auth flow under network failure, which is what break-it #2 covered. Two intentional breakages on this phase felt like the right depth.)
