# Phase 2 — Backend skeleton

## 1. Teach-back

Phase 2 turned the empty `backend/` corner of the repo into a process you can talk to. There is still no user, no auth, no data — just a server that answers `GET /health` with `{ok: true}`. The point of the phase isn't the route. It's everything *around* the route: the supply chain that makes one HTTP request a teachable trace through five layers, each with a clear job that the next layer trusts.

Trace a real request from the smoke test. `curl https://Santoshs-MacBook-Pro-48.local/health` first hits **Caddy** on port 443. Caddy is a separate process running under Homebrew's `launchd` because macOS won't let an unprivileged user bind ports below 1024 — `brew services start caddy` solves this once and forever, no sudo on each dev start. Caddy presents a TLS certificate generated by `mkcert` for that hostname; because mkcert's local CA is already in the macOS / iOS / Android trust stores from Phase 1, the curl client trusts it without `-k`. The same trust will carry to the iPhone and the Android Emulator in Phases 5 and 6 — no per-device cert dance. Caddy then strips TLS and forwards the plain HTTP request to `127.0.0.1:3000`, where the actual backend is listening.

The backend is **Bun** — a runtime that natively understands TypeScript without a build step, with a built-in web server, test runner, and SQL driver. **Hono** is the routing layer on top: a tiny TS-first framework whose strongest feature for this project is its testability, since the same `app.fetch` function that production hands to `Bun.serve` is what the unit tests hand to a synthesized `Request`. No port binding, no fetch mock, no integration harness — the HTTP transport is just a function call. That's why `routes.test.ts` can assert response bodies and headers in 50 ms.

Inside the app, every request hits the **request-ID middleware** before any feature code runs. The middleware reads an inbound `X-Request-ID` (validating it as printable ASCII to avoid log injection — newlines in a header would shred grep), or generates a UUIDv4 if absent. It builds a Pino *child logger* with `{reqId, method, path}` baked in, stores both the id and the child logger on the Hono context, and echoes the id back in the response header. Every subsequent log line in that request — including the central `onError` handler — emits the reqId for free, just by calling `c.get('logger')`. This is the single thread that ties together a curl invocation, a Caddy access log line, a Hono request log, and (in future phases) a Postgres query — without it, debugging concurrent requests degenerates into reading interleaved logs and guessing.

The **config module** is the second piece of plumbing. At module load time, it parses `process.env` against a Zod schema and freezes the result. A missing or malformed env var crashes the process before `Bun.serve` is ever called. This matters because the alternative — referencing `process.env.PORT` directly throughout the codebase — turns a typo into a runtime failure on first request, possibly in production. Pushing the failure forward to boot is the whole reason the typed config exists: a config that lies is worse than no config.

Phase 3 will add the database schema and start using `db.ts`. Phases 4+ will graduate the test suite from in-memory `app.fetch` calls to Testcontainers-backed integration tests. Everything those phases need — typed config, request-id-aware logging, a central error handler, a Hono app builder that's already test-friendly — is in place.

## 2. Rejected alternatives

**Decision**: Caddy under `brew services` (launchd-managed) reading a symlinked `/opt/homebrew/etc/Caddyfile`.
**Rejected**: `sudo caddy run --config backend/Caddyfile` for each dev session, and a `pf` port-forward 443→8443 with Caddy unprivileged.
**Why**: `sudo caddy run` adds a password prompt to every dev start and the Caddy process dies with the terminal — a small amount of friction repeated thousands of times. `pf` forwarding is sudo-free at runtime but requires more one-time setup and a launchd plist anyway to make the rule survive reboots. `brew services` is a launchd plist *with* the privileged-port story already solved by the formula. The trade-off is that the Caddyfile lives at an absolute path and the project file has to be symlinked, which is cosmetically ugly but a one-line README note.

**Decision**: `tls /absolute/path/dev.crt /absolute/path/dev.key` in the Caddyfile, certs generated with `mkcert` and stored under `backend/certs/` (gitignored).
**Rejected**: `tls internal` (Caddy's own embedded CA) and re-using the system mkcert default cert.
**Why**: `tls internal` makes Caddy generate certificates from its own CA, ignoring mkcert entirely. That CA is *not* in the macOS/iOS trust stores Phase 1 prepared, so iOS Simulator and physical phones in Phase 5 would refuse the connection and we'd be back to per-device trust dialogs. Pointing Caddy at mkcert-issued certs makes the device-trust story uniform across all phones from day one. The absolute paths are forced by the brew-services working directory (`/opt/homebrew/etc/`), not a choice.

## 3. Break-it session log

**What I broke (1)**: First Caddy run used `tls internal` and was started directly via `caddy run`.
**What happened**: Caddy bound :80 silently, *failed* to bind :443 (no error in stdout — just the absence of the listener), and tried to install its own root CA via `sudo`. The smoke test against `https://Santoshs-MacBook-Pro-48.local/health` couldn't connect at all.
**What I learned**: macOS unprivileged Caddy doesn't error loudly when it can't bind a privileged port; you have to check `lsof -iTCP:443 -sTCP:LISTEN` to notice. And `tls internal` is the wrong primitive when there's already a project-blessed CA — it's there for "just give me HTTPS without thinking," which is the opposite of this project's posture.
**What I changed**: rewrote the Caddyfile to use mkcert's `dev.crt`/`dev.key`, switched to `brew services start caddy`, and added a header comment to the Caddyfile explaining the launchd binding story so a six-months-from-now-me doesn't relearn this.

**What I broke (2)**: started the backend with `DATABASE_URL` set to a syntactically invalid string and watched config validation fire.
**What happened**: the process exited at module load with `Invalid environment configuration:` followed by Zod's tree of failures — *before* the Pino logger was constructed and *before* `Bun.serve` was called. Exit code 1.
**What I learned**: the noConsole Biome rule had to allow `console.error` precisely because the logger depends on the same module that crashes; if I'd kept noConsole strict, the only signal at boot-time failure would have been a silent exit. The chicken-and-egg here is genuine — this isn't a place for clever indirection.
**What I changed**: nothing in code. Updated the comment in `config.ts` to make the chicken-and-egg explicit so the next person reading this file doesn't try to "fix" the `console.error`.

**Skipped**: the PLAN-suggested "kill Postgres mid-request" experiment. Phase 2 has no Postgres-touching code path — Phase 3 introduces the first query, and that's the right phase for the kill-the-DB experiment. Logged here so it isn't lost.

## Appendix — JSON config rationale

JSON has no comment syntax, so the *why* behind individual config values lives here.

### `tsconfig.json`

- `target: ESNext`, `module: ESNext`, `moduleResolution: bundler`, `lib: [ESNext]`: Bun runs modern JS natively — no need to transpile down to ES2019 the way Node-CJS-with-tsc setups do. Bundler-style resolution lets us write `import { x } from './y.ts'` with the explicit `.ts` extension that Bun expects.
- `types: ["bun"]`: pulls in `Bun.serve`, `Bun.password`, etc. Without this, `Bun.serve` is `any` and we lose type checking on the server entry point.
- `strict: true` is the umbrella that turns on `strictNullChecks`, `noImplicitAny`, and several siblings. Required by the project's "never `any`" rule.
- `noUncheckedIndexedAccess: true`: makes `array[i]` typed as `T | undefined`, not `T`. Catches bugs where you assume an index exists. Costs a few `if (item)` checks but eliminates a class of runtime crashes.
- `noImplicitOverride: true`: requires the `override` keyword when a subclass method overrides a parent's. Prevents accidentally shadowing a method when the parent renames its method out from under you.
- `noFallthroughCasesInSwitch: true`: a `switch` case without `break` or `return` is an error. Rare bug source, almost free to enable.
- `exactOptionalPropertyTypes: true`: distinguishes `{x?: T}` (might be missing) from `{x: T | undefined}` (might be explicitly undefined). The Pino logger options needed this care — it's how the original `transport: undefined` setup was caught at compile time.
- `verbatimModuleSyntax: true`: forces `import type {...}` for type-only imports. Clearer for readers and avoids the bundler stripping a value-import that turns out to be type-only.
- `isolatedModules: true`: requires every file to be independently transpilable. Bun and modern bundlers assume this; turning it on catches cross-file type-only re-exports that would break under those tools.
- `allowImportingTsExtensions: true`: lets us write `from './foo.ts'` instead of `from './foo'`. Bun reads the `.ts` extension at runtime; matching it in source removes a guessing game for editors and IDEs.
- `noEmit: true`: tsc only typechecks; Bun does the actual execution. Without `noEmit`, tsc would emit `.js` files we don't want.
- `paths: {"@/*": ["src/*"]}`: optional aliasing for cleaner deep imports. Not used yet but prepared for when nested feature folders make `../../../` ugly.

### `biome.json`

- `vcs.useIgnoreFile: true`: Biome respects `.gitignore`, so it doesn't lint `node_modules` or generated `drizzle/` migrations.
- `lineWidth: 100`: a project-wide setting for prose-style line wrapping. Matches modern team norms (80 is too tight for TS generic types, 120 makes long-form comments hard to scan).
- `organizeImports: enabled`: auto-sorts imports on save/format. Eliminates a class of merge conflicts and bikeshedding.
- `noDefaultExport: error`: matches the project rule "named exports, not default." The override for `drizzle.config.ts` exists because Drizzle Kit *requires* a default export — that's the legitimate exception.
- `noConsole: { allow: ["error", "warn"] }`: bans `console.log` (use the structured logger) but allows `console.error`/`console.warn` for places where the logger isn't yet constructed (`config.ts` boot-failure path) or where a critical-warning predates logger setup.
- `noExplicitAny: error`: matches "never `any`" — TypeScript should always know the type, even if `unknown` is the answer.
- The `**/*.test.ts` override turns off `noConsole` because `console.log` in a failing test is sometimes the fastest way to debug. Production code has no such excuse.

### `package.json` scripts

- `dev: bun --watch src/index.ts`: `--watch` restarts the server on file change. Bun's watcher is process-level, so it picks up `.env` changes too.
- `start: bun src/index.ts`: production-style invocation, no watcher. Used by future Docker images.
- `typecheck: tsc --noEmit`: separate from `dev` because typechecking is slow (~1-2s on this codebase) and `bun --watch` doesn't need it on every restart.
- `db:generate / db:migrate / db:studio`: thin wrappers around Drizzle Kit CLI. Documented as scripts so the right one is just one keystroke away.
