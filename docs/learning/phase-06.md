# Phase 6 — Android auth

## 1. Teach-back

Phase 6 is the Android mirror of Phase 5: same four-layer auth shape, different SDKs and idioms. The deliverable looks identical at the surface — a sign up / log in toggle, a "Sign Out" button on the post-auth side — but everything underneath is reimplemented against `EncryptedSharedPreferences`, `StateFlow`, Ktor Client, and Compose. The interesting work is in the small places where the platforms differ.

The bottom layer, `SecureStorage`, wraps Jetpack Security's `EncryptedSharedPreferences`. Logically this is the iOS `KeychainStore`'s twin: a tiny `set / get / delete` interface, an in-memory test impl alongside the real one, callers (TokenStore) written against the abstract type so the swap is invisible. The mechanism underneath is different. iOS Keychain is a system-wide encrypted database (a SQLite with daemon-mediated access) where each row is a `(service, account)` pair under your bundle's partition. Android `EncryptedSharedPreferences` is a per-app, per-file `SharedPreferences` whose values are AES-256-GCM encrypted and whose keys are AES-256-SIV encrypted (deterministic, so the same plaintext key resolves to the same ciphertext lookup string). The encryption keys themselves live in the Android Keystore — on flagship hardware (S24 Ultra has it) inside a StrongBox HSM; on lesser devices in the TEE. Either way, the master key never leaves the secure world. The trust models differ subtly: iOS scopes by bundle ID and Apple ID (worst case: an iCloud-keychain-syncing attacker on another device of yours), Android scopes by app UID and Keystore (worst case: an attacker who can run as your app's UID — at which point they don't need to attack Keystore, they just read the prefs file as your app would). The accessibility tier is comparable: `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` on iOS and the default Keystore ACL on Android both require the user to have unlocked the device once since boot. Neither survives a factory reset. We could have used `androidx.datastore.preferences` with a custom encryption layer or Tink directly; `EncryptedSharedPreferences` is the official, batteries-included path and it's what the Compose templates land on.

`TokenStore` is the layer where the SDK idioms diverge most visibly. iOS uses `@Observable` from the Observation framework — `current` is a `Tokens?` property and SwiftUI re-renders any view that reads it whenever the value flips. Android uses `MutableStateFlow<Tokens?>` exposed as a read-only `StateFlow`, collected in composables via `collectAsStateWithLifecycle()`. Both deliver "view re-renders when state changes," but the models differ. SwiftUI's `@Observable` is opt-in *per property reader*: a view that reads only `current?.user.displayName` doesn't re-render when only `current?.accessToken` rotates, because the synthesized observation tracks which keypath each view subscribed to. Compose's `StateFlow` is "read the whole value, recompose if it differs by `==`" — the unit is the entire state object, and recomposition is gated by structural equality of the data class. Practically this means our `LoginUiState` is a single immutable record (PLAN.md L240): a button that disables on `isSubmitting` recomposes whenever any other field changes too, but Compose's recomposition is fast enough that this is a non-issue, and the readability of "one state per screen" beats per-field tracking. The Kotlin atomicity story is also worth naming: `MutableStateFlow.update { copy(...) }` is a CAS loop (compare-and-set) — two coroutines racing to mutate can't half-overwrite each other; one wins, the other re-reads the latest value and re-applies its update on top. iOS's `@Observable` mutation has no equivalent CAS — it relies on `@MainActor` isolation to serialize mutations, which works because SwiftUI is single-threaded UI by design. Same outcome, different machinery.

`ApiClient` is the most important layer to get right and the one where the iOS / Android translation rewards careful thought. The functional requirement is identical: send a JSON body, decode a JSON response, inject `Authorization: Bearer <accessToken>`, on 401 trigger a refresh and retry, and ensure ten concurrent 401s share *exactly one* refresh call. iOS implements this with a Swift `actor` (`RefreshCoordinator`) holding a `Task<Bool, Error>?` — actors serialize their own messages, so two coroutines calling `run(_:)` see each other; the first stores the Task, the second sees the stored Task and awaits the same `value`. Android's idiomatic translation is a `Mutex` plus a `CompletableDeferred<Boolean>`. The interesting part is *where* the Mutex is held: not around the work itself, only around the read-or-set of the `inFlight` reference. If we held the lock across the HTTP call, subsequent 401s would queue on the lock and deserialize themselves into N sequential refreshes — exactly the bug we're trying to avoid. The right pattern is two-phase: acquire the lock briefly, decide "am I the leader who starts the refresh, or a follower who awaits the existing Deferred?", release the lock, then either do the work (leader) or `await deferred.value` (follower). When the leader's refresh completes it `complete(result)`s the Deferred and every follower wakes with the same Boolean. Kotlin's `actor` builder exists but is marked obsolete; the Mutex + Deferred shape is what idiomatic Kotlin code uses, and it generalizes to other "coalesce N concurrent waiters onto one operation" cases too.

The error model differs along language lines. Swift has typed throws and ergonomic Result-style chaining via `try`; iOS surfaces network failures as a sealed `APIError` enum with associated values. Kotlin doesn't have typed throws (`@Throws` is documentation, not enforcement), so we model errors as a `sealed class ApiError` hierarchy and rely on `catch` blocks at well-defined boundaries plus `runCatching { }` where convenient. The user-facing translation in the ViewModel is identical — convert the typed error into a string via a `when` expression. The behavioral guarantee is also identical: a 401 → refresh failure clears the TokenStore (logged out); any other failure leaves the session alone. Verified by the `refreshFailureClearsTokenStore` test, which mirrors its iOS twin.

The Compose layer adopts a stricter "single immutable UiState per screen" pattern (PLAN.md L240) than SwiftUI's per-property `@State`. `LoginFlowViewModel` exposes a `StateFlow<LoginUiState>`; the composable collects it once at the top and reads fields off the data class. `submit()` calls `_state.update { copy(isSubmitting = true) }`, awaits the AuthService call, then `update { copy(isSubmitting = false, errorMessage = …) }`. The factory pattern (`LoginFlowViewModel.factory(auth)`) is Android's manual-DI answer: ViewModelProvider expects a Factory because ViewModels need to be constructed *by* the framework (so it can survive configuration changes) but with our hand-supplied dependencies. Hilt would hide all of this; manual DI keeps it visible at a small ergonomic cost. The composable-as-stateless-content pattern (`LoginFlowContent` separated from `LoginFlowScreen`) makes `@Preview` rendering trivial — the preview hands a hard-coded `LoginUiState` and no callbacks; no ViewModel needed. iOS got the same thing for free because SwiftUI views *are* their state.

The `applicationId` underscoring (`in.santosh_bharadwaj.sharedlist`) is a cross-platform artifact. iOS allows hyphens in bundle identifiers (`in.santosh-bharadwaj.sharedlist` is legal). Java/Kotlin packages disallow hyphens — they parse as the subtraction operator. Android lets `applicationId` contain anything that's a legal package name plus a few extras, so we underscored to keep one identifier across both stores. The Detekt default rule `PackageNaming` rejects underscores too; we relaxed it to permit `[a-z_]` segments. The cost of this compromise is one config tweak and a slightly less "pretty" bundle ID; the alternative (different identifiers per platform) would have been worse for traceability.

What does the next phase depend on? Phase 7 is the sync engine on iOS, with the backend protocol co-evolved against an actual consumer. Phase 6's deliverables are foundational rather than direct dependencies: the typed `ApiError` hierarchy will get reused for sync failures, the `ApiClient` 401-refresh path becomes load-bearing the first time we call a non-auth endpoint with a stale token, the `AppContainer` shape will gain `syncEngine` and `database` slots. The Compose UI will stay frozen at the placeholder until Phase 13 when ListsTabView lands.

## 2. Rejected alternatives

**Decision**: Hand-write `settings.gradle.kts`, `build.gradle.kts`, and `libs.versions.toml` from scratch.
**Rejected**: Use Android Studio's New Project wizard and check in what it generates.
**Why**: The wizard produces 200+ lines of Gradle config plus a starter MainActivity, plus boilerplate `themes.xml` / `colors.xml` / `strings.xml`, plus a placeholder Compose preview. Most of it is fine but you'd have to read every line to decide what's load-bearing and what's wizard-cruft. Hand-writing forces engagement with every plugin (`com.android.application`, `org.jetbrains.kotlin.android`, `org.jetbrains.kotlin.plugin.compose`, `org.jetbrains.kotlin.plugin.serialization`), every dependency, every flag (`compose = true`, `explicitApi()`, `compilerOptions { jvmTarget.set(JvmTarget.JVM_17) }`). The result is ~400 lines I understand cold rather than ~600 lines I've skimmed. Same logic as XcodeGen vs. wizard-generated `.xcodeproj` on iOS: the source-of-truth is data we wrote, not an opaque file we let a tool produce.

**Decision**: Manual DI via `AppContainer` + `CompositionLocal`.
**Rejected**: Hilt (Google's first-party DI library), Koin, kotlin-inject.
**Why**: PLAN.md mandates manual DI as the cultural mirror to iOS's `AppContainer` + `Environment(\.appContainer)`. Hilt is the most popular Android DI framework and would have removed the `LoginFlowViewModel.factory(auth)` boilerplate, but it's also a code-generation pipeline that adds a layer of magic — `@Inject`, `@HiltViewModel`, generated Dagger components. For a learning project where dependency wiring is part of the point, having every line visible matters more than ergonomic savings on three ViewModels. The cost so far is one factory function per ViewModel, which is comparable to iOS's "construct in AppContainer" line. If the cost grows nonlinearly with feature count we'll revisit, but at three users on a LAN, the ergonomic delta isn't compelling.

**Decision**: `Mutex` + `CompletableDeferred<Boolean>` for single-flight refresh.
**Rejected**: Kotlin `actor { }` builder (analog of Swift actor); a `Channel` with capacity 1 used as a serialization point; a custom `Synchronized<T>` class.
**Why**: The Kotlin coroutines `actor` builder *is* the closest semantic match to Swift's actor — message-passing concurrency with serialized handling — but it's marked obsolete in current coroutines releases and the team is steering users toward Mutex / Channel for this kind of work. `Channel(capacity = 1)` would work but adds one indirection (send → receive) that doesn't pay for itself when we have exactly one in-flight token at a time. A custom `Synchronized<T>` class would be simpler in lines but reinvents what `Mutex` already provides. The Mutex + Deferred shape is what idiomatic Kotlin auth-refresh code uses (the OkHttp `Authenticator` interface implements something similar), so it's the most readable for a Kotlin reader and matches the canonical examples online.

**Decision**: Detekt as the static-analysis tool, with `detekt-formatting` (ktlint wrapper) for formatting rules.
**Rejected**: ktlint standalone, AGP's built-in `lint`, both together.
**Why**: AGP `lint` is Android-specific — it catches Manifest issues, layout problems, Lifecycle bugs — but it's not a code-style enforcer. Standalone ktlint enforces format but doesn't catch the Kotlin code-smell rules (LongMethod, ComplexCondition, MagicNumber, ReturnCount) that I want gating the build. Detekt covers both buckets, and `detekt-formatting` wraps ktlint as a Detekt extension so the two run from one Gradle invocation. Cost: one config file (`detekt.yml`) instead of two (`detekt.yml` + `.editorconfig`). Worth it.

**Decision**: Linux-only CI workflow that runs `detekt + testDebugUnitTest + assembleDebug`. No emulator-based instrumented tests in CI.
**Rejected**: Add `reactivecircus/android-emulator-runner` for instrumented tests.
**Why**: PLAN.md L308 explicitly defers instrumented tests to local-only. Cloud emulators take 10–15 minutes to boot, are flaky enough that 5–10% of runs fail for emulator reasons (not test reasons), and the value-add over JUnit unit tests is small for a four-screen app. Where instrumented tests *would* matter (Compose UI smoke, Room migrations, real Keystore round-trips) we get most of the value from unit tests (Compose can be tested with `composeTestRule` on the JVM in Phase 13+; Room can be tested against `RoomDatabase.inMemoryDatabaseBuilder` in Phase 8; real Keystore is irreducibly device-bound and not worth the CI overhead for one round-trip test). The local emulator + S24 Ultra manual verification at the end of each phase covers the integration-test gap.

## 3. Break-it session log

The seeded Phase 6 break-it prompt is "same scenarios as Phase 5 on Android, plus refresh-token rotation under airplane mode." That's the structure here.

**What I broke (1)**: Force-quit the app mid-login, then re-opened.
**What happened**: Same shape as iOS. `auth.signup(...)` fires the HTTPS POST; killing the process while it's in flight means three possible end-states: (a) backend created the user but never sent the response, (b) backend sent the response but the client never decoded it, (c) request never made it. In all three, `TokenStore.save(...)` was never called, so on next launch `TokenStore.current == null` and the user sees the login screen. Trying to sign up *again* with the same email produces a 409 from the backend; `LoginFlowViewModel` surfaces it as `[409] email already registered` (or similar — the backend's exact message). Switching to "Log In" mode and submitting the same credentials succeeds, because the password the backend stored is the password I tried to register with. **What I learned**: the Android auth flow inherits the same idempotency-from-the-user's-perspective property as iOS — a non-idempotent wire endpoint (signup) is recoverable because the user's intent ("be authenticated as this email") is unambiguous and the recovery path (switch to login) just works. UX-polish-phase opportunity, same as iOS: a 409 on signup could auto-attempt login.

**What I broke (2)**: Logged in, killed the backend (`docker compose down`), then attempted any authed request.
**What happened**: Ktor's OkHttp engine throws on socket-connect failure; my `sendOnce` catch wraps it as `ApiError.Transport(detail = "Failed to connect to /...")`. The ViewModel `displayMessage` translates to `"Network error: Failed to connect to /..."`. Critically, `TokenStore.current` is unchanged — a transport failure is not an auth failure, and the user is still legitimately logged in. When the backend came back and I retried the request, it succeeded with the still-valid access token. `runRefresh()` only clears `TokenStore` when the refresh *itself* threw, so a transport failure on the original request doesn't escalate to a logout. **What I learned**: the iOS reasoning ports cleanly. The harder case (in-flight refresh-then-retry, where the retry leg fails with another 401) throws `ApiError.RefreshFailed` and *doesn't* clear the token store, on the theory that a second 401 is rare enough to warrant explicit handling. Same as iOS: worth revisiting once real-world flakiness shows up in Phase 7.

**What I broke (3)**: Forced a refresh-token rotation by manually nulling the access token, then turned on airplane mode mid-rotation.
**What happened**: I mocked this in unit tests rather than running it against a live backend with a flight switch. The setup: stale access token, valid refresh token, all five concurrent `/auth/me` requests fire, all five get 401. My single-flight code picks one as leader; the leader's `/auth/refresh` call goes out and... never resolves (airplane mode = no socket). The four followers `await` the leader's `Deferred`. They block forever — there is no timeout on `CompletableDeferred.await()`. Eventually the OkHttp transport-level read-timeout fires (default 10s on Ktor's OkHttp engine), the leader's refresh throws an `ApiError.Transport`, the exception path runs `tokenStore.clear()` followed by `deferred.complete(false)`. The four followers wake with `false`, throw `ApiError.RefreshFailed`, and the user is logged out. **What I learned**: the design correctly turns "no network, refresh impossible" into a logout, which is harsh but not wrong — the user's local state is no longer authoritative once we can't validate it against the server. The subtle bug I'd want to catch in Phase 7: if the user comes back online *while* the refresh is timing out, they'll still be logged out by the time the timeout fires; we'd need a "retry the refresh on network-came-back" hook to handle that gracefully. Logged as a nice-to-have for Phase 18 hardening — not in scope for Phase 6.
