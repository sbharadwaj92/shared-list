# Phase 10 — Backend WebSocket server + push infrastructure

## 1. Teach-back

Phase 10 introduces two pieces of plumbing the system has been waiting for since the sync engine landed: a WebSocket layer that pushes mutation events to interested clients in real time, and a push notification subsystem that delivers alerts to phones that aren't currently online. Both are infrastructure-only — Phases 11 and 12 connect the iOS and Android receivers; Phase 16 wires mutation events to the actor-excluding push fan-out. By the end of Phase 10 the backend speaks WS, knows how to register device tokens, and can deliver a push to APNs sandbox or FCM dev — what's missing is consumer code on the platforms and the per-event fan-out policy.

The WebSocket layer sits underneath Bun's native `Bun.serve`. The fetch handler at the top of `index.ts` does a path check: if the request is `/ws`, it tries to upgrade. Auth happens at upgrade time via a JWT-in-query-string (`?token=<jwt>`) rather than an `Authorization` header — browsers can't set headers on the WS handshake and the Authorization-header alternative would close off any future browser client without a protocol break. The query-string token leaks into access logs as a side effect; that's mitigated by the access token's 15-minute TTL and by Caddy's default `/ws` log format stripping the query string before persistence. If auth passes, `server.upgrade(req, { data })` attaches a typed `SocketData` blob (userId, connectionId, subscriptions Set) to the socket so subsequent `open`/`message`/`close` handlers can read it via `ws.data` without a side Map.

Once a socket is open, it sends typed messages: `subscribe`, `unsubscribe`, `ping`. The server replies with `ack`, `pong`, or `error`. Subscriptions are per-list — the client expresses interest in `list:<uuid>` topics, and Bun's native pub/sub (a uWebSockets-backed primitive) handles fan-out. Critically, `subscribe` re-checks membership against the database every time: the JWT proves you're authenticated but not that you're entitled to *this list's* events, and a revoked user must lose their subscription stream immediately. Production code paths and tests both go through `activeMembership(db, listId, userId)`, which filters `deleted_at IS NULL` on the member row.

The publish side is an `EventPublisher` seam. Mutation routes (lists/items POST/PATCH/DELETE) accept a `publisher` constructor argument and call `publisher.publish(event)` after each successful write. In production the publisher is `BunEventPublisher`, which calls `server.publish(topic, JSON.stringify(...))`; in tests it's `InMemoryEventPublisher`, which records published events in an array. The seam matters because route tests assert "this mutation produces this event with this listId" without standing up a real server. Event payloads carry `entity + action + id + listId + at` — never the row body. The shape is a freshness signal: receiving an event means "ask the server for fresh data via `?since=`," not "here's the new row." If the event body and the `?since=` cursor disagreed (out-of-order delivery, partial failure), the sync engine would have to do its own merge; deferring everything to reconciliation keeps the merge logic single-sourced.

The push subsystem is separate but parallel. `POST /devices` registers a (user, platform, token) tuple. The schema's unique index is on `token` alone — not `(user, token)` — because APNs and FCM tokens are globally unique by platform contract: when a user signs out and another signs in on the same physical device, the OS hands the new login the same token. If we used `(user, token)` as the uniqueness key, both rows would survive and the original user would keep getting the new user's pushes. The repo helper's `ON CONFLICT (token) DO UPDATE` semantics MOVE the row to the new owner, which is the only correct behaviour. The `id` field is client-generated UUID v7, idempotent on retries.

The senders speak to APNs HTTP/2 with `.p8` keys and FCM HTTP v1 with service-account JSON. Each sender is a closure-based "client" that caches its auth artifact (an ES256 JWT for APNs, an OAuth2 access token for FCM) and exposes `send(job)`. Job processing goes through pg-boss, a durable Postgres-backed queue: enqueue happens in the request path, processing happens in a worker callback on another tick. The worker translates the sender's `PushResult` (`ok | retry | unregistered | invalid`) into a queue-level action — `retry` throws so pg-boss applies backoff, `unregistered` returns normally and fires an `onUnregistered` hook (Phase 16 wires the cleanup), `invalid` logs and returns. The whole subsystem is opt-in: `PUSH_ENABLED=false` (the default) returns a no-op `DisabledPushService` so a fresh dev clone can boot without Apple Developer or Firebase credentials.

Putting it all together: a future mutation route in Phase 13 says "add item X to list Y." It writes the row, calls `publisher.publish({entity:'item', action:'created', id:X, listId:Y, ...})`, which fans out via Bun pub/sub to every subscribed iOS / Android socket. Then Phase 16 will call `pushService.notifyUser(memberId, {...})` for every member except the actor; the service enqueues per-device jobs in pg-boss, the worker drains them through APNs/FCM, and dormant phones get an alert. Phase 11 and 12 add the receivers that consume the WS events; Phase 16 layers actor-exclusion and "shared-list-only" gating on the push fan-out. The wire contract that landed today freezes both surfaces.

## 2. Rejected alternatives

**Decision**: JWT-in-query-string for WebSocket auth.  
**Rejected**: cookie-based auth, custom `Authorization` header on the upgrade request.  
**Why**: native iOS and Android can set headers on `URLSessionWebSocketTask` and Ktor WebSocket respectively, but the browser WebSocket API cannot — picking a header-based scheme would foreclose ever adding a browser client without a protocol break. Cookies work for browsers but neither URLSession nor Ktor has a default cookie jar; adding one for one use case is overkill. Query-string auth has the access-log leak risk, but our access token is short-lived (15 min) and Caddy's log format strips the query string by default on `/ws`. Industry convention for cross-platform mobile WS APIs has converged on query-string tokens.

**Decision**: WS events carry only `{entity, action, id, listId, at}` — never the row body.  
**Rejected**: include the new row in the event payload so clients can update local state without an extra `?since=` round-trip.  
**Why**: out-of-order delivery is real on a flaky network. If event N arrives after event N+1 carrying conflicting bodies, the receiving client would have to do its own LWW merge — duplicating the merge logic that already lives in the sync engine's `?since=` reconciler. Treating events as pure freshness signals (`receive event → trigger reconciliation`) means every code path that mutates local state goes through the same merge function, and the WS layer can't bypass it. We trade one extra HTTP round-trip per event for "the sync engine is the single source of truth for local state," which is the property that lets us reason about LWW correctness at all.

**Decision**: `EventPublisher` seam with separate `BunEventPublisher` and `InMemoryEventPublisher` impls.  
**Rejected**: call `server.publish()` directly from mutation routes.  
**Why**: routes that depend on the live `Bun.Server` are untestable in unit-style tests — every assertion would require booting `Bun.serve` on a port and opening a real WebSocket client. Routing event publishing through an interface (production: wraps `server.publish()`; tests: records in an array) means the route tests can assert "this mutation emits this event with this listId" in pure sub-100ms tests, and the WS integration tests (which DO use a real server) cover the publisher→subscriber end-to-end pipe separately. The cost is one extra layer of indirection on a method that's literally one line; the value is being able to refactor the routes without breaking the WS contract by accident.

**Decision**: device token uniqueness on `token` alone, with `ON CONFLICT (token) DO UPDATE`.  
**Rejected**: composite unique on `(user_id, token)`.  
**Why**: APNs and FCM tokens are globally unique by platform contract. When a user signs out and another signs in on the same physical device, the OS hands the new login the same token. Composite uniqueness would leave both rows in place and the original user would keep getting the new user's pushes — a privacy bug we'd discover only on the first dev handoff. Single-column uniqueness with ON CONFLICT UPDATE moves the row to the new owner atomically, which is the correct semantic.

**Decision**: pg-boss for the push queue.  
**Rejected**: hand-rolled `SELECT FOR UPDATE SKIP LOCKED` poller against a `push_jobs` table.  
**Why**: a hand-rolled queue is genuinely tempting in a learning project — the SQL is short and the queue concept is small. But durable retry, backoff schedules, dead-letter handling, and crash recovery are exactly the kind of thing where small mistakes manifest as "this push fires twice" or "this push silently never fires" in production. pg-boss has been kicking those tires for years. The learning value of writing the senders + the queue facade is high; the learning value of writing yet another at-least-once queue is low against the cost of getting the retry-on-process-restart story wrong. The facade (`PushQueue` interface, `InMemoryPushQueue` for tests) preserves the seam without paying for pg-boss inside unit tests.

## 3. Break-it session log

**Scenario 1: backend crashes mid-WS-connection**

- **What I broke**: kicked off `bun src/index.ts`, opened a WS connection via `bun /tmp/ws-verify.ts`, subscribed to a list. Sent SIGTERM to the backend mid-flight.
- **What happened**: the WebSocket's onclose fired with code 1006 (abnormal closure) within ~50ms. Subscriptions don't survive a restart because they live in process memory (Bun's pub/sub topic index isn't persisted) and `ws.data.subscriptions` is closure state. After restart, the client must reconnect and re-subscribe.
- **What I learned**: this is exactly the failure mode Phase 11/12's reconnect loop has to handle. The PLAN.md ordering — "pull `?since=` BEFORE re-subscribing" — exists because the time between disconnect and resubscribe is a hole where events were missed. Reconciliation closes the hole; re-subscribing without reconciling would silently lose events.
- **What I changed**: nothing yet — Phase 11 will implement the reconnect-with-reconcile semantics on the iOS side.

**Scenario 2: subscribe to a list after membership is revoked**

- **What I broke**: subscribed to list X, then went into psql and `UPDATE list_members SET deleted_at = now() WHERE list_id = X AND user_id = me`. Sent another `subscribe` for the same list.
- **What happened**: the second subscribe got back `{type:'error',code:'not_a_member',...}` because the dispatcher re-checks `activeMembership` every time. The first subscription (still open from before the revocation) keeps its topic membership in Bun's index — the dispatcher doesn't proactively unsubscribe revoked users because there's no event hook for `deleted_at` getting set. So a revoked user CAN still receive events on a list they're already subscribed to, until their socket closes.
- **What I learned**: this is a real gap. The current behaviour is "revocation takes effect on next subscribe or next reconnect." For Phase 16 it's fine — a revoked user re-opening the app reconnects, hits the membership check on subscribe, and falls off. But if revocation MUST be instant in some future use case, we'd need to either iterate `server.publish()`'s subscribers (Bun doesn't expose this) OR maintain our own `Map<userId, Set<ws>>` to do per-user disconnect. Not a Phase 10 fix; recording it as a known property of the protocol.
- **What I changed**: documented this in `backend/docs/sync.md` under WS revocation semantics.

**Scenario 3: malformed JSON over an open WS**

- **What I broke**: `ws.send('{ this is not valid json')`.
- **What happened**: the dispatcher caught the `JSON.parse` failure, sent back `{type:'error',code:'invalid_message',message:'message is not valid JSON'}`, and kept the socket open. The next valid message (a `ping`) got a `pong` back as normal.
- **What I learned**: the handler-level try/catch is correct — a single bad client doesn't close the socket. If we DID close on bad input, a buggy mobile client would reconnect-storm and DoS itself. The error message is enough for the developer to debug; for production we'd want a counter, but at 3 users that's overkill.

**Scenario 4: subscribe to 100 lists simultaneously (Bun pub/sub stress)**

- **What I broke**: extended the verification script to create 100 lists and subscribe to all of them on one socket.
- **What happened**: all 100 subscribes returned `ack`s; the WS layer didn't choke. Then triggered a mutation on every list; all 100 events arrived in <50ms. Bun's pub/sub is a C++ implementation under uWebSockets and is *very* fast — this is the property that makes the architecture viable at all (a JS-level `Map<topic, Set<ws>>` would have stuttered on the fan-out).
- **What I learned**: subscription-density is not the bottleneck at our scale. The bottleneck for real-world performance would be DB queries on each `subscribe` (one `activeMembership` per subscribe) — those go through the connection pool and at 100 concurrent subscribes from one client we'd hit pool exhaustion before we hit WS limits. Not a Phase 10 concern at 3 users.

**Scenario 5: PUSH_ENABLED=true with malformed APNs PRIVATE_KEY**

- **What I broke**: set `PUSH_ENABLED=true`, set all the APNs env vars, but put garbage (`"not a pem"`) into `APNS_PRIVATE_KEY`. Booted the backend.
- **What happened**: the boot succeeded (config validation only checks `min(1)` length, not PEM validity). The first `notifyUser` call would have thrown from `importPKCS8`, propagated through the worker as a transport throw, and pg-boss would have retried 5 times before giving up. There's no boot-time check that the key is parseable.
- **What I learned**: this is the kind of "permanently broken config that retries forever" footgun pg-boss is supposed to protect against. The current behaviour is "5 retries then dead-letter" which is fine — the operator sees the dead-letter accumulating, looks at logs, sees the `importPKCS8` failure, fixes the key. For a production system I'd add a boot-time `importPKCS8(APNS_PRIVATE_KEY)` smoke that fails fast on a bad key. For this learning project: documented in `backend/README.md` under "if pushes never arrive, check pg-boss dead-letter."
- **What I changed**: added a brief diagnostic step to `backend/README.md`. Not adding a boot-time PEM smoke for now (the retry-then-dead-letter path is itself worth observing).

The break-it session surfaced one real protocol property (revoked-but-still-subscribed users keep their subscription until disconnect) that I documented but didn't fix, and one ops-discoverability gap (bad APNs key produces silent-ish dead-letters) that I noted in the README. The remaining scenarios confirmed expected behaviour, which is itself the point — the learning is in noticing whether the behaviour matches the mental model, not in fixing every observation.
