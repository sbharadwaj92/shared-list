# Phase 8 — Android persistence + sync engine

## 1. Teach-back

Phase 8 is mirror-work. Phase 7 froze the wire protocol — `?since=` cursors, `If-Match` conditional writes, idempotent `POST` with client-generated UUIDs, soft-delete tombstones — and the iOS slice ran the engine through a slice-by-slice design: read-side reconciler (B), mutation queue (C.2), drainer with 409→reconcile→retry-once (C.3), and a fuzz wrap (D). Phase 8's job was to port the same shape to Kotlin / Room / Ktor without redesigning anything. The interesting part is where the platforms genuinely differ; the rest ports one-to-one.

The biggest platform difference is the persistence layer. SwiftData on iOS gives you `@Model` reference types where every read goes through `ModelContext.fetch(FetchDescriptor)` and every write batches into one `context.save()`. Room on Android gives you `@Entity` data classes (immutable Kotlin records) and `@Dao` interfaces whose methods are SQL queries — `@Upsert`, `@Query("SELECT ... WHERE ...")`, `@Delete`. The "atomic batch" primitive is `database.withTransaction { … }` (an extension on `RoomDatabase` shipped in `room-ktx`), which serializes everything inside the lambda into one SQLite transaction. The translation is direct: every iOS `try context.save()` block becomes a `database.withTransaction { … }`, every iOS `FetchDescriptor` becomes a `@Query` annotation, every iOS LWW upsert becomes a "find existing row, compare `updatedAt`, upsert" sequence inside a transaction. Schema sizes match: six entity types (User, List, Item, Member, SyncCursor, MutationQueue), one composite-key table (Member uses Room's native `primaryKeys = ["listId", "userId"]` — cleaner than the iOS pipe-joined string workaround for SwiftData's missing composite-unique support).

The second platform difference is JSON. Foundation's `JSONEncoder.dateEncodingStrategy = .iso8601` is second-precision by default — that was the slice-D iOS bug. kotlinx.serialization is a different shape: `Instant` has *no* built-in serializer, so the user must supply one explicitly. The path of least resistance ("use a `String` field plus `Instant.parse(...)` at the call site") works on the read path but loses type safety. The real trap — the one I deliberately wrote a test for *before* it could surface — is that `Instant.toString()` truncates trailing zero fractional digits: an instant constructed from `…56.000Z` returns the bare-seconds form `…56Z`, dropping the fractional component entirely. Sending that as `If-Match` to a backend that has `…56.000Z` on disk is the same 409 storm waiting to happen. The fix is `JsonCoders.kt` with an explicit `InstantIso8601MillisSerializer` that emits exactly three fractional digits via `DateTimeFormatter.ofPattern("uuuu-MM-dd'T'HH:mm:ss.SSS'Z'")` and accepts both fractional and bare-seconds forms on parse. Five `JsonCodersTest` assertions pin the four properties that matter: writes always emit three digits even for `.000`, writes preserve non-zero millis, sub-millis truncate (don't round), the round-trip is exact, and bare-seconds parses forward. This was the single load-bearing piece — getting it right up front meant slice D didn't surface a Kotlin equivalent of the iOS Foundation-default bug.

The third platform difference is concurrency. iOS lives on `@MainActor` for everything that touches SwiftData; the drainer hops to a `Task { @MainActor }` after a network call. Kotlin lives on coroutines — `suspend` functions implicitly hop to whatever dispatcher the caller picked, and Room's IO is its own internal dispatcher (you don't need `withContext(Dispatchers.IO)` around a suspend DAO call). The drainer owns its own `CoroutineScope(SupervisorJob() + Dispatchers.IO)` — `SupervisorJob` so a failure in one drain coroutine doesn't cancel siblings, `Dispatchers.IO` because both Ktor and Room want IO threads. The single-flight semantics that iOS gets from `actor` come from `Mutex.withLock { }` plus a `CompletableDeferred<Boolean>` for the in-flight handle. Same guarantee as iOS's `RefreshCoordinator` actor (one coroutine holds the work; followers `await()` the same `Deferred`), different syntactic shape.

The reactive-state shape also differs. iOS uses `@Observable` plus `NWPathMonitor` for connectivity; SwiftUI tracks the boolean property by keypath. Android uses `StateFlow<Boolean>` plus `ConnectivityManager.NetworkCallback`, and Compose subscribes via `collectAsState()`. The `NetworkMonitor` registers a `NetworkRequest` filtered to `NET_CAPABILITY_INTERNET + NET_CAPABILITY_VALIDATED` so we don't call ourselves online behind a captive portal. We track every active network in a `MutableSet<Network>` (multi-network is real on Android — Wi-Fi + cellular concurrently when one is metered) and treat "the set is non-empty" as the online signal. The set is intentionally not exposed; only the derived boolean is, because the rest of the app doesn't need to know the network type.

The fuzz tests are the same four hostile scenarios as iOS — rapid create-delete-create, simultaneous edits with chained `If-Match`, edit-on-server-deleted, and list-deleted-while-items-have-pending-mutations — and they catch the same kinds of bugs. One Kotlin-specific subtlety: under `runTest` with a `TestScope`, every `kick()` queues a coroutine in the same scope, and suspension points inside Mutator calls (the Room DAO awaits) yield to those kick coroutines. That means a fuzz test that pre-builds the queue and *then* calls `drainer.tick()` explicitly will see the queue partially drained before the assertion runs if the Mutator was wired with `attachDrainer`. The fix in `SyncFuzzTest` is to deliberately *not* call `mutator.attachDrainer(drainer)` — production wires both ends, but fuzz tests build the queue first and tick explicitly. iOS doesn't hit this because Swift Tasks don't run mid-call between sibling Mutator invocations on `@MainActor`. The `DrainerTest` suite still covers the attached-drainer path through unit-style scenarios.

The integration tests use the same env-gated convention as iOS: `BACKEND_URL` flips them on, `JUnit.assumeTrue` makes the unset case a "skipped" status (not a failure). PLAN.md L380's "real backend" requirement is satisfied by a backend that's actually running, regardless of who started it. Keeping the convention symmetric across platforms means the same dev loop on either side: backend runs continuously, app + tests target it. CI integration is deferred — same reasoning as iOS in commit 65f1305, just with the Android quirk that Linux runners *could* boot Postgres + Bun (no nested-virtualization wall like macOS) but the symmetry-with-iOS argument won. Revisit in Phase 19 polish.

What does the next phase depend on? Phase 9 is cross-platform sync verification — a test harness that drives both iOS and Android engines against the same backend and asserts the four scenarios converge consistently. The wire protocol is now genuinely frozen on both sides; the only remaining work is the harness itself.

## 2. Rejected alternatives

**Decision**: One single PR for all four slices (B' → C.2' → C.3' → D').
**Rejected**: Four sequential PRs mirroring iOS Phase 7's cadence (#8, #10, #11, #12, #13).
**Why**: The design was already known from Phase 7. Phase 7's per-slice PRs were necessary because the design was being figured out in slice-sized chunks — slice B's reconciler shape informed C.1's write-side, which informed C.2's mutation queue. Phase 8 had no such design discovery; the answer to "how does the Mutator look" was "exactly like iOS, in Kotlin." Splitting into four PRs would mean four review rounds covering the same one-to-one porting decisions. The owner asked for one PR and accepted the reviewer-cost trade-off (one large diff vs four small ones) for the development-velocity win (no waiting for review at slice boundaries when there's nothing genuinely new to discuss).

**Decision**: Robolectric for the Room-touching unit tests (`@RunWith(RobolectricTestRunner)`, `@Config(sdk = [35])`).
**Rejected**: Wrap Room behind a `Repository` interface and use an in-memory map implementation in tests.
**Why**: Robolectric adds a dependency and makes tests slightly slower (Robolectric 4.14 boots a fake Android runtime per test class). The cost is real but bounded. Wrapping Room behind a hand-rolled repository interface would force every DAO method to have a corresponding interface method, a corresponding map-based implementation, and a corresponding contract-validation test. That's three artifacts per DAO method instead of one annotation. Worse, the "in-memory map" test impl wouldn't exercise the real SQLite SQL and wouldn't catch SQL syntax errors (e.g. a typo in a `@Query` annotation) until production. Robolectric's `Room.inMemoryDatabaseBuilder(context)` runs the *real* Room compiler output against the *real* SQLite, so test failures point at real bugs. The trade-off was: spend a one-time tax (Robolectric setup + a `@Config(sdk = [35])` annotation per Room test class) for the gain of testing the actual persistence path. We took the tax.

**Decision**: Explicit `InstantIso8601MillisSerializer` for kotlinx.serialization on every `Instant` field that crosses the wire.
**Rejected**: kotlinx-datetime's `Instant` (which has built-in ISO-8601 serialization).
**Why**: Two reasons. First, Room 2.6's `TypeConverter` story is smoother for `java.time.Instant` than for kotlinx-datetime types — Java time interop is documented, kotlinx-datetime requires custom converters. Second, mixing two `Instant` types in one codebase (`java.time.Instant` everywhere except the wire layer, kotlinx-datetime's `Instant` only on the wire) introduces a translation step at every boundary. The 30-line custom serializer keeps the model coherent: `java.time.Instant` end-to-end. The serializer is ~40 lines including format constants, decoder, encoder, and `companion`/`object` boilerplate. That's cheaper than the alternative.

**Decision**: Don't `attachDrainer` to the Mutator in `SyncFuzzTest` (build queue first, tick explicitly).
**Rejected**: Wire the drainer normally and use `Dispatchers.Unconfined` or some other scheduler-control to keep `kick()` coroutines from running between Mutator calls.
**Why**: Under `runTest` + `TestScope`, every `kick()` queues a launch in the same scope. The Mutator's suspending DAO calls yield to those launches at suspension points, so by the time the test asserts "the queue has 6 entries", some of them have already been drained. Trying to fight this with dispatcher tricks (e.g., `Dispatchers.Unconfined` for the drainer's scope) would undermine the production behavior the test is supposed to be verifying. The cleaner solution is to recognize that the fuzz tests are doing two things — verifying the Mutator builds the queue correctly and verifying the Drainer drains it correctly — and to seam the two phases explicitly. `DrainerTest` covers the attached-drainer path through unit-scoped scenarios where the queue build + tick happen as one observed sequence. iOS dodges this because Swift Tasks don't yield mid-call on the same `@MainActor`; the difference between the two languages' coroutine semantics costs us this one-line "don't attachDrainer in fuzz fixtures" workaround.

**Decision**: Same env-gated integration-test convention as iOS (`BACKEND_URL` env var, `JUnit.assumeTrue` skip).
**Rejected**: Wire up an Android integration CI workflow that boots Postgres + Bun on the Linux runner before running `./gradlew testDebugUnitTest --tests '*Integration*'`.
**Why**: Linux runners on GitHub Actions *can* boot Postgres + Bun (unlike macOS, which couldn't nest-virtualize for the iOS attempt — see commit 65f1305 for that history). We technically had the option to break symmetry and wire CI integration for Android only. We didn't, for two reasons. First, dev-loop symmetry: same instructions on both platforms ("`bun run dev`, then `BACKEND_URL=… ./gradlew test`"), same skip behavior on missing env. Two paths for the same conceptual thing makes the project harder to reason about. Second, scope: this phase's goal is "Android sync engine works against the live backend," not "Android integration CI works." The latter is a Phase 19 polish, scheduled when we revisit iOS integration CI under "self-hosted macOS runner or paid Mac CI." Wiring one platform's CI now would give half the value (only Android-side coverage) and leak design choices into the eventual two-platform CI implementation. Better to defer.

## 3. Break-it session log

Phase 8 was mirror-work, so there were fewer surprises than Phase 7 — the load-bearing decisions had already been made. Two notable ones:

**What I broke (1)**: SyncFuzzTest's three timing-sensitive scenarios on the first test run.
**What happened**: `rapidCreateDeleteCreateConverges` expected 6 queue entries (3 create + 3 delete) before draining, got 5. `simultaneousLocalEditsDrainInOrder` had If-Match values that didn't strictly increase. `listDeletedWhileItemsHavePendingMutationsConverges` expected 2 queue entries (item PATCH + list DELETE), got 1. The pattern across all three: a queue entry was missing or already drained before the assertion ran.
**What I learned**: Under `runTest` with a `TestScope`, every `kick()` from the Mutator launches a coroutine in the same scope. The Mutator's suspending DAO calls yield to those kick coroutines at suspension points, so a kick from `createList` can drain a queue entry between the next `deleteList` call and the assertion. iOS doesn't hit this because Swift Tasks scheduled with `Task { @MainActor }` don't run mid-call between sibling Mutator invocations on the same actor. Kotlin's coroutine model is more eager about yielding, and `runTest`'s deterministic scheduler means "yields between every suspension" is the rule rather than the exception.
**What I changed**: The fix was structural, not a flag tweak. `SyncFuzzTest.environment(...)` now deliberately does NOT call `mutator.attachDrainer(drainer)`. The fuzz tests build the queue with all entries first, then call `drainer.tick()` explicitly. Production wires both ends through `attachDrainer`; the `DrainerTest` suite (where queue build and tick are unit-scoped to the same scenario) covers the attached path. A four-line comment in the fuzz fixture explains the rationale so a future contributor doesn't "fix" it by re-adding the attach. This pattern translates the iOS test design to Kotlin while respecting the language's stricter-yield semantics.

**What I broke (2)**: Detekt's `MagicNumber` rule on the Drainer's HTTP status branches (`200..201`, `500..599`).
**What happened**: First detekt pass after wiring slices C.3' + D' flagged six magic-number warnings on the status-code ranges. The rule is on by default and reasonable in general — `200..201` doesn't tell a reader "successful 2xx range from OK to Created."
**What I learned**: Detekt's bias toward "name everything" is the right call for a learning project. The Drainer was already importing `HttpStatusCode` from Ktor for some checks but using bare integers for the ranges, which is exactly the inconsistency lint should catch.
**What I changed**: Extracted six named constants in the Drainer's `companion object` (`HTTP_OK`, `HTTP_CREATED`, `HTTP_NO_CONTENT`, `HTTP_UNAUTHORIZED`, `HTTP_FORBIDDEN`, `HTTP_NOT_FOUND`, `HTTP_CONFLICT`, `HTTP_INTERNAL_SERVER_ERROR`, `HTTP_NETWORK_AUTH_REQUIRED`) plus two named ranges (`SUCCESS_2XX = HTTP_OK..HTTP_CREATED`, `SERVER_5XX = HTTP_INTERNAL_SERVER_ERROR..HTTP_NETWORK_AUTH_REQUIRED`). The branch sites now read `in SUCCESS_2XX` and `in SERVER_5XX`, which is exactly what the iOS code says (Swift's switch case `case 200, 201:` is self-documenting; Kotlin's `200..201` isn't). Cheap fix, materially better readability.
