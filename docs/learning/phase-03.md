# Phase 3 — Backend schema + migrations

## 1. Teach-back

Phase 3 turned the empty Postgres database that Phase 2 was pointed at into a real schema — seven tables, every column typed, every relation enforced, and one trigger that quietly underwrites the entire sync engine that comes later. There is still no API beyond `/health`, no auth, no users to insert. The point isn't reads or writes; the point is that the *shape* of what we will read and write later is now committed, can be migrated forward, and is verified by tests against an actual Postgres rather than a mock.

The schema lives in one file: `backend/src/infra/schema.ts`. Drizzle's TypeScript DSL describes each table — `pgTable('lists', { ... })` — and Drizzle Kit reads that file and generates SQL migrations from it. Phase 3 produced two migrations: `0000_romantic_bromley.sql` (the table DDL Drizzle Kit auto-generated) and `0001_updated_at_triggers.sql` (a hand-written follow-up). When you run `bun run db:migrate`, Drizzle Kit walks `drizzle/meta/_journal.json`, applies migrations in order, and records each one in a `__drizzle_migrations` table inside the database itself. That last bit matters: the migration runner is idempotent because it checks the journal table before applying, so re-running it on a database that's already current does nothing.

The seven tables map onto the v1 product cleanly. `users` holds login identities. `lists` is the unit of sharing — note that even the creator is *not* a foreign-key on the list itself, only a row in `list_members` with `role = 'owner'`. That uniformity simplifies authz for every endpoint to "is the caller a member?" rather than "is the caller the owner OR a member?". `items` belongs to a list and carries the integer `position` we'll sort by. `device_tokens`, `list_invites`, and `refresh_tokens` exist for the auth/push flows in later phases; they are scaffolded now so we generate one big migration instead of seven small ones, and so the FK shape across the whole schema is consistent.

The cross-cutting design choices are the part worth remembering: **UUID v7 primary keys generated in app code, never by the database.** The IDs are time-ordered (good for B-tree locality) but more importantly they let the iOS and Android clients generate IDs offline, so a `POST` retried after a network blip uses the same `id` and the server's `INSERT ... ON CONFLICT DO NOTHING` makes the retry safe. This is the whole story behind the sync engine's idempotency, and it falls out of the schema decision rather than being a separate feature later.

**Soft delete on `lists`, `items`, `list_members` only**; the other four tables don't carry a `deleted_at`. Soft delete is sync-protocol-internal: a deleted list still has to flow to other clients during the 90-day retention window so they can drop it from their UIs, which means `?since=` reads will surface tombstones. Repo helpers (`activeLists`, `activeItems`, `activeMembership`) wrap every read with `WHERE deleted_at IS NULL`. The two places allowed to read directly are the future 90-day purge job and the future `?since=` endpoint. Anywhere else, raw access is a bug — and the layer that makes that rule visible is the existence of these named helpers in the first place.

**`updated_at` is set by a Postgres trigger, not by application code.** This is the load-bearing piece that most surprised me on review. Drizzle has an `$onUpdate` hook, but it only fires when the change goes through Drizzle. Any SQL that touches a row through `psql`, a future migration backfill, or admin maintenance would silently bypass it. The trigger function (`set_updated_at()`) and per-table `BEFORE UPDATE` triggers in `0001_updated_at_triggers.sql` make "any row change bumps the timestamp" a database-level guarantee, which the sync engine's last-write-wins comparison depends on.

Verification is via Testcontainers — `setupTestDatabase()` boots a real `postgres:17-alpine` container per test file, applies the same `drizzle/` migrations the dev DB uses, and lets the repo tests assert against actual constraints (FKs reject orphaned items, the functional `lower(email)` index rejects case-variant duplicates, the trigger really does bump `updated_at`). Phase 4 inherits this: when auth tests need a fresh DB, they call `setupTestDatabase()` and they're done. There is no test-only fake of the schema; the schema *is* the test target.

## 2. Rejected alternatives

**Decision**: A hand-written `0001_updated_at_triggers.sql` migration alongside the Drizzle Kit-generated `0000_*.sql`, with a manually-edited `_journal.json` entry and a duplicated `0001_snapshot.json` (since the trigger doesn't change anything Drizzle Kit tracks).
**Rejected**: Using Drizzle's `$onUpdate(() => new Date())` column hook on every table.
**Why**: `$onUpdate` only fires for writes that go through Drizzle — a future migration that runs raw SQL, a `psql` admin session, a `pg-boss` job that issues a `client.query`, all silently bypass it. Last-write-wins is only as correct as the timestamp that drives it, so "trust the application layer" is the wrong altitude for this guarantee. The cost is one extra hand-written migration file plus the awkward dance of pretending a snapshot exists for it; the benefit is "any UPDATE bumps `updated_at`" becomes a property of the database, not a property of the path the write took.

**Decision**: `postgres-js` driver added solely so `drizzle-kit migrate` and the test harness work; production runtime stays on `bun-sql`. The repo `Database` type is widened to `PgDatabase<PgQueryResultHKT, Record<string, unknown>>` so a single helper accepts either driver.
**Rejected**: Switching the runtime to `postgres-js` for uniformity, or running migrations through a custom Bun script that uses `bun-sql` directly.
**Why**: Drizzle Kit's migrator only ships node-style adapters; `bun-sql` is not one of them, so day-of-Phase-3 you either install `postgres-js` or you reinvent the migrator (parse `_journal.json`, compute applied vs. pending against `__drizzle_migrations`, run the SQL files yourself). Reinventing the migrator is a learning rabbit hole that buys nothing. Switching the runtime to `postgres-js` would leave behind one of the things this project specifically picked Bun for. The middle path — keep `bun-sql` in production, use `postgres-js` for migrations and tests, type the repo against the abstract base — pays one type-system cost (the wider `Database` type) for keeping both drivers' best properties.

## 3. Break-it session log

**What I broke (1)**: Bypassed `activeItems()` with `SELECT id, text, deleted_at FROM items WHERE list_id = ...` directly via `psql`, on a freshly-seeded list with one live and one soft-deleted row.
**What happened**: Saw both rows. The live one, and the tombstone with `deleted_at = 2026-05-04 20:28:53.91729+00`. Then the same SELECT with `AND deleted_at IS NULL` returned only the live row.
**What I learned**: This is the failure mode the repo helpers exist to prevent, and the data they hide is *exactly* what the sync engine needs to read. Two different consumers, same column, opposite semantics — and `db.select().from(items)` doesn't enforce a default. The only thing keeping feature code honest is the named helper. There is no compile-time guard against a future contributor writing the raw form; the code review and the test that asserts `activeItems` filters tombstones are what I have. Made me appreciate why this needs to be in `LEARNING_PROTOCOL.md`-grade memory rather than just a CLAUDE.md note.
**What I changed**: Nothing in code; the structure is correct. Added the rejected-alternative entry above so the future-me debugging a "why is my list count off" bug remembers to grep for raw `db.select().from(items)` first.

**What I broke (2)**: Tried to manually backdate `updated_at` on an existing row with `UPDATE items SET text = 'rewind attempt', updated_at = '2020-01-01T00:00:00Z' WHERE id = ...`.
**What happened**: The trigger overrode the manual value. The row's `updated_at` became `2026-05-04 20:29:24.159444+00`, not `2020-01-01`. The text update succeeded; the time-rewind did not.
**What I learned**: This is a stronger guarantee than I'd have gotten from `$onUpdate`. Even an attacker (or a future me being clever) cannot construct an UPDATE that lowers `updated_at`. That property is what makes `?since=<lastSync>` correct — the cutoff can never be evaded by a backdated write. Worth more than I expected from a four-line plpgsql function.
**What I changed**: Strengthened the comment in `0001_updated_at_triggers.sql` to make explicit that the trigger overrides any application-supplied `updated_at`, since the property isn't obvious from the SQL alone.

**Skipped**: The PLAN-suggested "run the daily purge job and observe what's deleted." The purge job is Phase 18+ work; nothing to manually run yet. Logged here so it isn't lost — the experiment to do once the job exists is "purge with a row whose `deleted_at` is 91 days old vs. 89 days old, observe both before and after, ensure the boundary condition is right."
