# Phase 4 — Backend auth

## 1. Teach-back

Phase 4 turns the schema from Phase 3 into a real authenticated API. There are now four endpoints — `POST /auth/signup`, `/login`, `/refresh`, `/logout` — plus a small `/auth/me` for clients to read their own user, and Swagger UI at `/swagger-ui` driven by an OpenAPI spec that's generated from the same Zod schemas the handlers use to validate. The schema you see in the docs and the schema that rejects bad input at runtime are the same object; there is no parallel YAML file to keep in sync.

The mental model: there are two token types in this system, and they look identical at the wire-shape level but mean opposite things. The **access token** is a signed JWT (HS256, 15-minute TTL). It is *stateless* — the server doesn't remember it, doesn't have a "list of valid access tokens" to consult. Every request that needs auth carries one in `Authorization: Bearer <jwt>`, and `requireAuth` middleware verifies the signature and reads the `sub` claim to know which user is talking. If the access token expires, the server says 401 and the client moves on to step two. The **refresh token** is the opposite: an opaque random string, *stateful*, stored in a DB row indexed by sha256 of its cleartext. The client keeps the cleartext (Keychain on iOS, EncryptedSharedPreferences on Android in later phases). The DB has only the hash. So if someone steals the database, they get hashes — not usable bearer credentials.

The interesting algorithm in this phase is **rotation with reuse detection** on `/auth/refresh`. The client posts its refresh token. The server hashes it, looks up the row by hash, and checks `used_at`. If `used_at IS NULL`, this is a legitimate refresh: mark the row as used (atomic CAS — the UPDATE is gated on `used_at IS NULL`, so two concurrent refreshes have exactly one winner), insert a fresh row with a new random token, and return both a new access token and the new refresh token. The client overwrites its stored refresh token with the new one. Everything is single-use; nothing lives long. If `used_at` is already set when the row is looked up, that's the panic path: someone is replaying a token whose first legitimate use already happened. Either an attacker has it, or a buggy client racing two refreshes does. We can't tell which, so the response is the same — DELETE every `refresh_tokens` row for that user. They (and any other devices they're logged into) are forced to re-login within the access-token TTL. The blast radius of a stolen refresh token is bounded to 15 minutes by construction.

Why 15 minutes specifically? Because that's the maximum window during which a stolen-but-not-yet-replayed access token gives the attacker access without showing up in the rotation. The refresh-token side handles the long tail; the access-token TTL caps the worst case. Make it 1 minute and every other request triggers a refresh; make it 24 hours and a stolen token is dangerous for a day.

The single-flight-refresh requirement — which lives on the *client* side, in iOS APIClient and Android ApiClient in later phases — is the missing piece this phase deliberately leaves to the consumers. The pattern is: when an in-flight request returns 401, the first 401 starts a refresh; every subsequent 401 awaits the same in-flight refresh promise. Without single-flight, two parallel requests both expire, both see 401, both call `/auth/refresh` with the same refresh token, both succeed at the lookup, and one of them loses the CAS — the loser triggers reuse detection and nukes the user. This is why backend reuse detection and client single-flight are joined at the hip: backend is hostile to replay, so client must never replay.

The HTTP plumbing is mostly Hono with one wrinkle. The auth feature uses `@hono/zod-openapi`'s `OpenAPIHono`, which extends plain Hono with a route registry that emits an OpenAPI 3.1 document. We mount that at `/openapi.json` and the Swagger UI at `/swagger-ui`. `/health` stays on plain Hono — it has no schema worth documenting. Mixing the two is fine because `OpenAPIHono` *is a* `Hono`. Validation, error handling, request-ID tracing, structured Pino logs all flow through the same paths Phase 2 set up. The auth service throws `HTTPException` for every expected failure; the central `onError` shapes everything into the same `{error: {code, message, requestId}}` envelope, so a client never has to string-parse or branch on response shapes between 401-from-validation and 401-from-bad-credentials.

Tests live in three layers, and which layer a test belongs in is the design question. Unit tests for `password.ts` and `tokens.ts` cover the cryptographic primitives in isolation — fast, no DB. Service tests in `service.test.ts` exercise the business logic against a real Testcontainers Postgres but skip HTTP: they call `signup()` directly, assert on row state, and verify the reuse-detection nuke at the data layer. Integration tests in `integration.test.ts` go through `app.fetch` end-to-end — Zod validation, status codes, the OpenAPI spec generation, the `requireAuth` middleware. There's also a tiny rate-limits test that builds a separate app with limits enabled (the main integration tests turn them off because `hono-rate-limiter` keeps process-wide in-memory state and the 3-per-hour signup cap would 429-out the third test case). Forty-five tests pass against real Postgres in under nine seconds.

## 2. Rejected alternatives

**Decision**: `buildApp(db, opts)` and `buildAuthRoutes(db, opts)` accept the database and an options object as parameters; integration tests pass a Testcontainers DB and `enableRateLimits: false`.
**Rejected**: Using the module-level singleton `db` directly inside the auth routes (the obvious first cut), and gating rate-limiter middleware on `process.env.NODE_ENV === 'test'`.
**Why**: The singleton couples the route file to one specific Postgres connection, so swapping in a Testcontainers DB requires monkey-patching the module or stubbing `infra/db.ts`. Threading `db` through as an argument mirrors what the repo helpers already do (Phase 3) and keeps the dependency explicit. The `NODE_ENV` gate would have worked but it's a runtime side-channel: tests would silently get different middleware than production, and a forgotten `NODE_ENV=test` in CI (or its absence locally) would mask a real bug. An explicit `enableRateLimits: false` makes "this test runs without rate limits" obvious in the test file itself, and a dedicated `rate-limits.test.ts` enables them and verifies the 429 — coverage is preserved, the indirection isn't.

**Decision**: HS256 (symmetric) JWTs, with the secret in `JWT_SECRET` env var, validated by Zod at config load (minimum 32 chars).
**Rejected**: RS256 (asymmetric) with a keypair in PEM files; or a key file in `secrets/` outside the repo.
**Why**: HS256 is correct for this topology — there's a single backend process that both signs and verifies. RS256 buys you "the verifier doesn't need the signing key," which only matters if you have a *separate* verifier service. We don't, and adding a keypair just to learn the asymmetry is exactly the kind of speculative complexity PLAN.md warns against. The Zod `min(32)` is the part that mattered: it makes "the secret was too short" a boot-time failure, not a "your tokens silently get weak HMACs" runtime failure. We can revisit RS256 later — the JWT alg is one of three places in `tokens.ts`, all marked, and the code already verifies via `algorithms: [ALG]` so a future swap is a one-line change plus a key-management decision.

## 3. Break-it session log

**What I broke (1)**: Replayed a refresh token over real TLS via `curl`, with two devices logged in for the same user. Used `breakit-1777928117@example.com` for the test account; took a snapshot of both refresh tokens after signup + login.
**What happened**: First refresh of device A's token → HTTP 200 with a new pair of tokens. Replay of the *same* device A token → HTTP 401 with body `{"error":{"code":"http_exception","message":"refresh token reuse detected — re-login","requestId":"..."}}`. Then attempted device B's refresh (which had never been used) → HTTP 401 with `"invalid refresh token"`. Confirmed at the DB level: `SELECT count(*) FROM refresh_tokens WHERE user_id = ...` returned 0. The revoke-all really does delete every row.
**What I learned**: The two error messages are intentionally different at the *server* level (`"reuse detected"` vs `"invalid refresh token"`) but the client cannot distinguish them — both are 401 and both lead to "send the user back to the login screen." The differentiated server message is for the logs and the request-ID-cross-reference, not for the client. Also: the legitimate device-A refresh and the second device-B login both came back with the **same** access token. They were minted within the same second, with the same `sub`, and JWT `iat`/`exp` resolution is one second. Two HMACs over the same payload yield the same string. This is fine cryptographically, but it surprised me — the access token is not a per-issuance unique identifier. Worth knowing for any future log analysis that tries to count "tokens issued."
**What I changed**: Nothing in code. Added a note in this learning doc instead — the second-resolution `iat` is a JWT property, not a bug, and "fixing" it would mean adding a random `jti` claim that nothing else in the system looks at.

**What I broke (2)**: Sent malformed bearer headers to `/auth/me`: `NotBearer foo`, `Bearer`, `Bearer ` (trailing space, empty token), `Bearer  abc.def.ghi` (double space).
**What happened**: All four returned 401 with the same `"missing"` or `"malformed"` message, depending on which guard caught them. The regex `^Bearer\s+(.+)$/i` accepts the double-space case (matches more than one whitespace), and `verifyAccessToken` then rejects the bogus JWT shape with a different error path. Empty token after "Bearer " → `match?.[1]?.trim()` is the empty string after trim, and the `if (!token)` branch catches it cleanly.
**What I learned**: The optional-chained array index access (`match?.[1]?.trim()`) catches both "regex didn't match" and "regex matched but capture group was empty after trim" with one expression. Without `noUncheckedIndexedAccess` in tsconfig, TypeScript would have happily let `match[1]` through and the runtime guard would have been a `if (typeof token !== 'string')` afterthought. The strict tsconfig setting paid for itself here. Also: I didn't add a separate test case for *every* one of the malformed headers above, only one (`NotBearer foo`). The other variants are covered by the integration suite happening to exercise them in passing. If I wanted to be thorough I'd add four more parameterized cases, but the regex is small and the test value is marginal.
**What I changed**: Strengthened the comment in `auth.ts` middleware to spell out *why* `match?.[1]?.trim()` looks paranoid — it's the `noUncheckedIndexedAccess` interaction, not a defensive coding instinct.
