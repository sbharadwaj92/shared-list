# Phase 7 — Backend sync protocol + iOS sync engine in tandem

## 1. Teach-back

Phase 7 is the spine of this project. Phases 1–6 produced a backend that knows how to authenticate and a pair of mobile apps that know how to log in — useful but inert. Phase 7 makes the system *act* like a synchronizing app, with a wire protocol on the backend and a sync engine on iOS that's ambitious enough to actually exercise it. The phase landed across four slices (A, B, C.1, C.2, C.3, D) because the design only makes sense end-to-end and a single PR would have been impossible to review.

The sync protocol's central problem is "two devices, occasional connectivity, no central source of truth except the server, and no merge UI." We solved this by splitting the contract into three pieces and making each piece independently testable.

The first piece is **the read feed**: `GET /sync/lists?since=<ISO8601>`, `/sync/items`, `/sync/list_members`. Every endpoint returns rows whose `updated_at > since`, *including soft-deleted rows*. The "including" is the load-bearing word. A naïve sync feed that returns only live rows can't tell a client "this list disappeared" — the row simply stops appearing, which is indistinguishable from "you've already seen it." Surfacing soft-deleted rows as tombstones (with `deletedAt: ISO8601`) lets the client drop the local copy on the next pull. The cost is keeping deleted rows around for a 90-day retention window, which is fine for a list app.

The cursor itself is `serverTime`, captured by the server *before* the SELECT runs, and echoed back to the client. The "before, not after" detail matters: a write that commits during the SELECT is invisible to that snapshot but its `updated_at` is necessarily later than the captured `serverTime`. The next pull (`since=serverTime`) catches it. Capturing AFTER the SELECT would lose rows whose `updated_at` falls between SELECT-end and `now()`-call. We trade a possible duplicate for a guarantee of no missed rows. Duplicates are idempotent on the client; missed rows are silent corruption. Postgres `now()` returns microseconds, but we `date_trunc('milliseconds', now())` everywhere because Date in JS / Swift / Kotlin is millisecond-precision and unrounded microseconds round-trip lossily through clients. A trigger normalizes every `updated_at` write to ms, and the cursor matches.

The second piece is **the write side**, slice C.1. Each mutation endpoint has two protocol-shaping properties: idempotent POST and conditional PATCH. Idempotent POST means the client generates the row's UUID (v7 in spec, v4 in our current iOS impl — see rejected alternatives), and the server does `INSERT ... ON CONFLICT (id) DO NOTHING`. A network-blip retry hits ON CONFLICT, returns 200 (vs 201 first time), and the client sees the canonical row from the original request. Without client-generated ids, every retry would create a new row. Conditional PATCH means the client sends `If-Match: <updated_at>` and the server CASes against the current row's `updated_at`. On match: 200 with the new row. On mismatch: 409 with `{error, latest: …DTO}` so the client can fold the divergent state in without an extra GET. We picked 409 over RFC 7232's 412 because PLAN.md L62 wanted one status for every "your write didn't land because of state divergence" case, regardless of whether it's id-collision or stale-cursor. DELETE soft-deletes by setting `deleted_at`; for lists it cascades to items in the same transaction so each item gets its own `updated_at` bump and surfaces on the items feed as a tombstone.

The third piece is **the iOS sync engine**, which is where the protocol actually has to work. Slice B's `SyncEngine` consumes the read feed and reconciles into a SwiftData store: lists → items → members in that order, because a self-revocation tombstone on the members feed sweeps the local list and any orphan items, and that sweep needs to happen *after* any leftover rows have landed (otherwise the items pull would just re-introduce them). LWW guards every upsert: a wire row with `updatedAt` ≤ local is ignored, so an unrelated re-pull doesn't clobber a fresh local mutation.

Slice C.2 introduced the **mutation queue**. Every user action (`createList`, `renameList`, `deleteList`, `createItem`, `patchItem`, `deleteItem`) goes through a `Mutator` that does two things in one SwiftData `save()`: applies the change to the local store optimistically, and appends a `MutationQueueEntry` row capturing the intent. The "in one save" is the load-bearing detail. If the local apply succeeded but the queue append didn't, we'd silently lose a write to the backend (looks correct locally, never reaches the server, eventually erased by reconcile). One save commits both rows atomically, and the SwiftData container persists the queue across app restart, so a force-quit between mutate and drain doesn't lose anything. The mutator pre-stamps local `updatedAt` to `clock.now()` and persists the *prior* `updatedAt` as the payload's `If-Match` — the value the server still has, not the one we just wrote.

Slice C.3 introduced **the drainer**, which closes the loop. It polls `pending` queue rows in `createdAt` order, decodes the payload, sends the right HTTP request, deletes the row on success. A single `isDraining` flag and a per-tick "stop on requeue" rule keep it from spinning. Status handling is mostly mechanical: 2xx removes, 404 removes (idempotent — server already gone), 403 marks failed (membership lost), 5xx and network errors re-queue with `retryCount++` for the next kick. The interesting case is 409 on PATCH. The body carries `latest: …DTO`; the drainer applies it through the SyncEngine's existing LWW upsert (the same code the reconciler runs), re-reads the now-merged local row, and sends one retry with the new `If-Match`. A second 409 means another device is rapidly editing the same row — we mark `failed` rather than spin, because PLAN.md L195's "LWW is the v1 conflict-resolution strategy" doesn't require us to build merge UI for an edit war.

LWW deserves a beat of explanation because it's the conflict-resolution strategy that's "not correct in the strict sense." The data-loss scenario: device A's edit at t=100 and device B's edit at t=101 both touch the same field, and both go through the queue + drainer. B's edit lands second, so the server stores B's value with `updated_at=t101`. A's earlier edit is silently lost — no merge prompt, no retry, no surfaced conflict. For a 3-user grocery list, this is fine: the chance of two users editing the same item's text simultaneously is small, and "the most recent person wins" matches everyone's intuition. For a collaborative document editor it would be a non-starter; for our scope it's the simplest correct strategy.

A real bug that surfaced in slice D: `Date.formatted(.iso8601...)` *truncates* the fractional component instead of rounding. A Date whose `timeIntervalSince1970` is `9000.001999...` (which is how floating-point represents `9000.002`) renders as `02:30:00.001` instead of `02:30:00.002`. The chained `If-Match` values for two rapid Mutator calls collapsed onto the same string, which would have produced 409 storms in production. We switched to `ISO8601DateFormatter` (which rounds correctly) via a shared `JSONCoders` module that emits + parses fractional seconds end-to-end. This is the kind of bug a unit test that just checks "do dates encode" wouldn't catch — the fuzz scenario "three edits in quick succession" was what surfaced it.

What does the next phase depend on? Phase 8 is the Android mirror. The wire protocol is now frozen; Android consumes the same `?since=`, `If-Match`, idempotent-POST contract through Ktor + Room. The `Mutator` / `Drainer` shape ports cleanly because the trade-offs we made (serial drain, optimistic-apply-with-queue, LWW upsert) aren't iOS-specific. The interesting Android-only work will be where Room differs from SwiftData and where Kotlin's coroutines + `StateFlow` differ from Swift's `@MainActor` + `@Observable`.

## 2. Rejected alternatives

**Decision**: Cursor as `serverTime` captured server-side before the SELECT, returned alongside the rows, and echoed back as the next `since`.
**Rejected**: Have the client compute the next cursor as `max(updated_at over received rows)`.
**Why**: Two failure modes. (1) Empty pull: no rows came back, no max — the client now has to fall back to its own clock or persist the previous cursor verbatim. Either path leaks complexity into every client. (2) Precision drift: Postgres `timestamptz` natively stores microseconds; JS Date / Swift Date / Kotlin Instant are millisecond. A client receiving `updated_at` and echoing it back loses sub-millisecond bits, and `>` filtering then re-streams the at-the-cursor row. Server-vouched `serverTime` (truncated to ms via `date_trunc`) keeps everyone on the same precision and lets empty pulls work without special-casing.

**Decision**: 409 on If-Match mismatch (body carries `latest: …DTO`).
**Rejected**: 412 Precondition Failed per RFC 7232 §4.2; 200 with a "merge required" envelope; 409 with no body and force the client to GET.
**Why**: PLAN.md L62 picked 409 explicitly. The deeper reason is that idempotency-id collision (POST with a tombstoned id) and stale-cursor (PATCH with old If-Match) are the same shape of failure from the client's perspective: "your write didn't land because the server's state diverged from yours." Putting both on 409 means the client has one branch to handle ("did the server reject for state reasons?") instead of two (412 for stale, 409 for collision). RFC purity loses to client simplicity. Returning the `latest` row in the body skips a follow-up GET and lets the merge happen in one round trip.

**Decision**: Client UUIDs are v4 (`Foundation.UUID()`) on iOS for now.
**Rejected**: Implement a UUID v7 generator (PLAN.md L47 prefers v7 for index locality on the server side).
**Why**: The backend's `INSERT ... ON CONFLICT (id) DO NOTHING` only validates uniqueness; version bits don't matter for idempotency. v7's win is time-ordered ids that are friendlier to Postgres B-tree index locality on `lists.id` / `items.id`. For 3 users on a single Mac, that win is unmeasurable. Implementing a real v7 generator (correctly handling the timestamp + monotonic counter + random tail) is its own sub-project. Treated as a Phase 19 polish; we accept the small index-locality loss for v1. The known-debt note lives inline in `Mutator.swift` so future me can find it.

**Decision**: Env-gated integration tests (`BACKEND_URL` env var) rather than self-bootstrapping Bun + Testcontainers from Swift.
**Rejected**: Spawn Bun + docker-compose Postgres from `Process` calls in `setUp`, like the backend's own Testcontainers tests do for Postgres-from-Bun.
**Why**: PLAN.md L380 only requires "real backend", not "test self-bootstraps the backend." Booting Bun + Postgres from Swift adds material CI fragility (DOCKER_HOST detection on macOS runners, Bun install path, port collisions, lifecycle on test crash, log tailing). The env-gated approach matches how iOS dev already runs locally — backend runs continuously in another terminal, app talks to it — and the CI workflow that boots backend in a step before running `xcodebuild test` is a few lines of YAML rather than a Swift `Process`-management module. The cost is two paths instead of one (CI workflow + the test code), but each path is small.

**Decision**: Mutation queue persisted in SwiftData alongside the cached models (same `ModelContainer`).
**Rejected**: UserDefaults; a separate SQLite database; a flat-file JSONL queue.
**Why**: SwiftData wins because the local-apply + queue-append must be atomic. If they're in different stores, a crash between them leaves an inconsistent state — local row updated but no queue entry (silent loss of the write), or queue entry without the local row (drainer sends a request the user "didn't take"). One `context.save()` across both `@Model` types is the simplest possible atomicity primitive and SwiftData gives it to us for free. UserDefaults can't carry structured data well; a separate SQLite duplicates the persistence layer; flat-file JSONL needs hand-rolled fsync semantics. The persistence shape doesn't dominate the value — the fact that BOTH writes commit or NEITHER does does.

## 3. Break-it session log

The seeded Phase 7 break-it: airplane mode → make 5 mutations → kill app → reopen → re-enable network. Plus a "during in-flight request" interruption.

**What I broke (1)**: Inside the slice-D fuzz suite, three rapid `renameList` calls within the same wall-clock millisecond.
**What happened**: First time the test ran, the recorded `If-Match` headers across the three resulting PATCHes all collapsed to the same string. Two requests with the same If-Match sent serially is a hard 409 from the server (the second can't match because the first just bumped `updated_at`). Tracing: Mutator's `RenameListPayload.ifMatch` was being encoded via Foundation's `JSONEncoder.dateEncodingStrategy = .iso8601` — which wraps an `ISO8601DateFormatter` with the default `formatOptions = .withInternetDateTime` (second precision only). Every Date going to disk was rounded to the nearest second. Even worse: when the Drainer formatted the (still-correct in-memory) `Date` via `Date.formatted(.iso8601.year().month()...)`, the format-style API silently TRUNCATES the fractional component instead of rounding, so `.001999...` (the IEEE-754 representation of `.002`) renders as `.001`.
**What I learned**: Two distinct millisecond-precision bugs. The first is a Foundation default that's wrong for any wire protocol carrying sub-second timestamps. The second is a `Date.formatted(...)` quirk I didn't know about — `ISO8601DateFormatter.string(from:)` rounds correctly but the format-style API doesn't. In retrospect the read-feed contract had been documenting "millisecond-precision" since slice A; the gap was that no test exercised the *write* side's roundtripping until slice D's fuzz tests made the precision matter for two consecutive patches.
**What I changed**: Wrote a `JSONCoders` module exposing `makeEncoder()` / `makeDecoder()` that use a shared `ISO8601DateFormatter` with `.withFractionalSeconds`. Replaced every `dateEncodingStrategy = .iso8601` and `Date.formatted(.iso8601...)` call with the shared coder / formatter. APIClient, Mutator, Drainer, SyncEngine. Backend was already correct (Postgres `date_trunc('milliseconds', now())` + JSON.stringify's automatic ISO emission with millis), so this was a one-sided iOS bug. The fuzz test that surfaced it stays in the suite as a regression guard.

**What I broke (2)**: 5xx-only response loop without a per-tick stop on requeue.
**What happened**: Drainer's `tick()` was a `while monitor.isOnline { takeNextPending; drain }` loop. The Drainer-test scenario "5xx response" requeues the row at `pending` for the next attempt. Without a per-tick break, the loop immediately picked up the same row again, sent another HTTP request, the MockSession's queue was exhausted, the request threw `unscriptedRequest` (wrapped as `transport`), the catch path requeued, the loop picked it up again... infinite loop. The xcodebuild test runner caught it as a "Restarting after unexpected exit" rather than a stack overflow because the leak was in the iteration count, not the call stack.
**What I learned**: A "drain until empty" loop has to distinguish "this row succeeded and was removed" (next iteration picks the next row) from "this row was requeued for later" (we should stop and wait for the next kick). Without that distinction, transient failures mean re-attempting the same row immediately, which is at best wasted requests and at worst infinite loops.
**What I changed**: Added an `entryStillPending(id:)` check at the bottom of each loop iteration. If the row we just attempted is still at `pending` (the requeue path), break out of the tick and wait for the next kick. This also matches the user's expectation: "I'll try again later" should mean LATER, not "immediately five times more."

**What I broke (3)**: Mid-cycle simulated server-deletes-while-client-edits race in the slice-D fuzz suite.
**What happened**: I queued an item PATCH locally, then before the drainer ran I tombstoned the same item on the server (scripted MockSession returning 404 to the PATCH). The drainer's first `case 404:` arm just removed the queue row. I half-expected to need extra reconcile machinery to sweep the local row too, but the existing read-feed reconciler already handles that — the server's tombstone will surface on the next `?since=` items pull and `deleteLocalItem` runs. The PR scope question was "should the drainer ALSO sweep on 404 to make the local state immediately consistent?" — and the answer is no. Two reasons: (a) the reconcile path is the authoritative cleanup, and duplicating its work in the drainer would mean two code paths that have to stay in sync; (b) the brief window where the local row exists but is destined to be tombstoned doesn't cause user-visible inconsistency because feature views (when they exist) read through helpers that filter `deletedAt == nil`, and the next reconcile is at most one foreground tick away.
**What I learned**: The instinct "fix everything immediately" can produce duplicated code; sometimes the right move is to trust the next regularly-scheduled reconcile to catch up. The trade-off is well-modeled because the contract for both layers (drainer / reconciler) is about convergence, not immediate consistency.
**What I changed**: Nothing in the code — confirmed the existing 404-path-removes-queue-row + reconcile-sweeps-local-row composition was correct. Added a comment in the `drainPatchItem` 404 branch noting that the reconciler is the authoritative sweep and the drainer's job is just to retire the queue entry. Left the fuzz test in place as a regression guard against future "let me also sweep on 404" temptations.
