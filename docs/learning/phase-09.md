# Phase 9 — Cross-platform sync verification

## 1. Teach-back

Phase 9 is the proof. Phases 7 and 8 built two independent sync engines — one Swift / SwiftData / `URLSession`, one Kotlin / Room / Ktor — both targeting the same wire contract that landed in Phase 7 slice C.1. Each platform had its own integration tests (iOS `DrainerIntegrationTests`, Android `DrainerIntegrationTest`), but those only verified "this platform's engine round-trips a mutation through the backend." Phase 9 asks the harder question: "do the two platforms' engines, acting as two devices owned by the same user, converge to identical local state under the four scenarios PLAN.md L386 calls out?" If the answer is yes, the wire-protocol contract is genuinely frozen and any future divergence can be attributed to an explicit protocol change.

The harness shape that emerged was Option 2 from the pre-coding discussion — a shell driver that invokes each platform's env-gated test suite per "act" or "observe" half of a scenario, threading list/item ids forward via env vars and structured stdout markers. The alternative shapes were considered and rejected: Option 1 (shell hits the backend HTTP API directly to simulate device A, then runs only the platform-specific reconciler) doesn't actually exercise the cross-platform encoding path because device A never produces real bytes from one platform that device B then decodes; Option 3 (each platform runs both A and B in-process plus one mixed handoff for parity) is fewer test invocations but loses the property that every scenario tests both directions. Option 2's cost is shell coordination — the harness is ~400 lines of bash, parses test stdout via grep, and threads state across processes via env vars — but the property it buys is exactly what PLAN.md asks for: two real processes, two real `ApiClient`/`TokenStore` instances, two real local stores, one shared backend.

The key constraint that shaped the harness was `POST /lists/:id/invites` not existing yet. PLAN.md keeps the invite/accept routes in Phase 15 because they pair with sharing-flow UI; without them, two backend users can't actually share a list. The four PLAN.md scenarios *don't* require different users — they require two devices — so the harness signs up one user at start and has both platforms `login` from independent `ApiClient`/`TokenStore` pairs. This is the right scope for "is the sync engine correct?" because sharing is a Phase 15 feature, and the membership-revocation tombstone path (the most subtle bit of the read-side reconciler) genuinely needs a second user, which we can revisit in Phase 15's testing. The two-devices-one-user pattern still exercises every sync mechanism: `?since=` cursors, `If-Match` conditional writes, idempotent POST, soft-delete tombstones, the drainer's 409→reconcile→retry-once dance, LWW upsert.

Each platform got a `CrossPlatformConvergenceTest{s}` suite with one method per role per scenario plus a `setup_seedFreshList` neutral seed step the harness invokes before scenarios B/C/D so each scenario starts from a known list. Tests skip silently when env vars aren't set (iOS uses guard-and-return, Android uses `assumeNotNull` / `assumeTrue`) so plain `xcodebuild test` and `./gradlew testDebugUnitTest` runs see them as 9 pass-with-nothing on iOS and 9 skipped on Android. The harness invocation always sets the env vars; running tests outside the harness is meaningless and silently no-ops.

The harness ran into three failure modes during development, each instructive:

The first was the iOS `-only-testing` syntax. XCTest accepts `BundleName/ClassName/methodName`; Swift Testing accepts `BundleName/StructName/methodName()` — the trailing `()` distinguish a single zero-arg test from a parameterized test family. Without them xcodebuild silently runs zero tests *and reports the test run as passed*. The first harness invocation passed every scenario "successfully" with zero assertions executed. That is a long-standing footgun in Swift Testing's xcodebuild integration; the only signal that something is wrong is the "Test run with 0 tests in 1 suite passed" line. The fix is to add `()` to the `-only-testing` paths in the harness.

The second was Android's stdout capture. Gradle suppresses test stdout by default — a `println` from a JUnit `@Test` goes into the per-class JUnit XML's `<system-out>` element rather than the gradle log. The harness's first cut grepped the gradle log for `CROSS_PLATFORM_RESULT[KEY]=value` lines and found none. The fix is to extract `<system-out>` from `app/build/test-results/testDebugUnitTest/TEST-*.xml` after the gradle run and feed it to the same grep. This is a reasonable design choice on Gradle's part (test stdout would otherwise drown the build output) but it's surprising the first time you hit it. The harness's `run_android` function now does both: tees the gradle log for human consumption, and reads the JUnit XML for parse-able results.

The third was a real test bug, not infrastructure: scenario (d)'s observer initially asserted `assertNotNull(item)` plus `assertNotNull(item.deletedAt)`, expecting the read-side reconciler to *upsert a tombstoned row*. The actual contract is "delete the local row entirely" — `SyncEngine.reconcileItems` calls `itemDao.deleteById(id)` (Android) or `deleteLocalItem` (iOS) on a tombstoned wire row, and on a row that was never seen locally that's a no-op. The end state is "no local row" regardless of whether the device had ever seen the item. The assertion got flipped to `assertNull(item)`, which is the correct contract. This is the kind of bug Phase 9 exists to catch: the contract is implicit in the code, the test made it explicit.

The convergence assertion for scenario (c) is the most architecturally interesting. Both devices rename the same list, against the same If-Match cursor (because both reconciled from the same seeded state). One write lands clean (200), the other 409s, reconciles, and retries-once. The harness's first cut compared `act_final` (the act platform's FINAL_NAME *immediately* after its own drain) against `observe_final`, but `act_final` was captured *before* the observer's later rename hit the server, so they didn't match — the test reported "FAIL" even though both platforms eventually-converge. The fix is a third step: a `scenarioC_reconcileOnly` invocation on the act platform after the observer ran, which pulls the now-final canonical row. Convergence is then "act_post_refresh == observe_final." This three-step shape is what eventual consistency actually looks like under a sequential test driver — both writes have to commit, both reads have to refresh, before you can compare. Real-world UI would refresh continuously via WebSocket events (Phase 11+); the harness simulates that by re-reconciling explicitly.

The bigger question — where the iOS and Android engines might genuinely diverge in observable behavior — was not the question I expected to discover. The visible divergence the harness found was scenario (d)'s "tombstoned row never previously seen": both platforms behave identically (delete = drop the local row, no insert), so my expectation that "tombstone = upsert with `deletedAt` set" was wrong on both platforms simultaneously. The harness *agreed* with itself across platforms; the test was where the expectation lived.

What does this say about the design? The wire-format contract is doing most of the work. The backend's `?since=` feed is platform-agnostic by construction: it returns full row snapshots, not deltas, and the merge logic on each client is "match by id, LWW on `updatedAt`, drop if tombstoned." Two implementations of that logic in different languages produce identical observable behavior because the merge function is referentially transparent given identical inputs. The non-trivial divergences are on the encoding side — Foundation's default `.iso8601` second-precision (caught in Phase 7 slice D), kotlinx.serialization's `Instant.toString()` zero-truncation (front-loaded in Phase 8) — and both have explicit fixes in `JSONCoders.swift` / `JsonCoders.kt`. Phase 9's value isn't finding a new divergence; it's *proving the divergence-prevention worked*.

What does Phase 9 not cover? The harness runs scenarios sequentially per pair; "true concurrent" edits (two writes within the same millisecond from different platforms) aren't reproducible from a shell driver. The backend's `set_updated_at` trigger truncates to milliseconds (Phase 7 slice A's fix), so two writes within the same millisecond would collapse to the same timestamp on the server, which the protocol explicitly handles via UUID v7 ordering as a tiebreaker. The fuzz tests in `SyncFuzzTests` / `SyncFuzzTest` use `AdvancingClock` to force the same-millisecond case in a single-platform setting; cross-platform same-ms fuzz is a polish item if we ever observe a real production divergence. For PLAN.md L386 ("two devices' sync engines provably converge"), the harness as built is sufficient.

## 2. Rejected alternatives

**Decision**: Option 2 — shell harness driving each platform's env-gated test suite for one role per scenario invocation.
**Rejected**: Option 1 — shell hits backend HTTP API directly, simulating device A, then runs only the platform reconciler.
**Why**: Option 1 would never produce real platform-encoded bytes — the encoder asymmetry between Foundation `.iso8601` and kotlinx.serialization `Instant.toString()` could only be caught by actually invoking each platform's `Mutator` to write to the backend and the *other* platform's read path to consume those bytes. Option 1's shell driver is faster and simpler but tests an artificially clean wire path. Option 2 spends time (~10 minutes for a full 8-pair run) for the actual cross-platform encoding parity guarantee, which is the load-bearing property Phase 9 exists to verify.

**Decision**: Option 3 also rejected — each platform runs both A+B in-process, plus one mixed handoff for parity.
**Why**: The in-process variant lets the test cheat with shared in-memory state — both A and B can see each other's local rows even without HTTP. Option 3 strictly speaking mostly tests the local merge logic, not the wire path. The mixed-handoff sub-step would catch encoding asymmetries, but only for a single scenario direction. Option 2's "every scenario in both directions" property is what makes a future regression in either platform's encoder show up as a scenario failure rather than as a silent corruption that survives because the in-process path didn't exercise the bytes.

**Decision**: Same-user / two-device pattern (sign up one user at harness start, both platforms log in independently).
**Rejected**: Multi-user / shared list (sign up two users at start, share list A→B).
**Why**: The sharing flow's `POST /lists/:id/invites` and `POST /invites/:code/accept` routes are Phase 15 deliverables. Pulling them forward to enable Phase 9's testing would be plan deviation without strong cause — PLAN.md L386's four scenarios *don't* require different users (they require two devices), and the membership-revocation tombstone path (the only scenario that genuinely needs a second user) is best tested when the routes that enable sharing actually exist. Same-user / two-device exercises every sync mechanism the four scenarios describe and stays inside Phase 9's scope cleanly.

**Decision**: Bump rate limits permanently (signup 3/h → 10/h, login 5/m → 30/m) before harness work.
**Rejected**: Add a `BACKEND_DISABLE_RATE_LIMITS` env hatch.
**Why**: An env hatch ships a backdoor that exists only for tests. The owner's preference was "permanent generous rate limiting" — the limits exist as a learning artifact (demonstrating the request → 429 → retry-after pattern), not as a real defense layer for a 3-user local-LAN deployment. Generous-but-not-infinite is the right balance; the dedicated `rate-limits.test.ts` suite still pins the throttle behavior at the new numbers (10 succeed, 11th returns 429). PLAN.md L81 updated in lockstep so the rate-limits.ts comment can keep pointing at the plan as the source of truth.

**Decision**: One PR for the whole phase.
**Rejected**: Slicing per-platform or per-scenario.
**Why**: Same reasoning as Phase 8. Phase 7 was sliced because the *design* was being discovered slice-by-slice; Phase 9's design was conversational up front (two harness-shape options + one user-vs-two-users decision, all settled before code). The implementation is mirror-work between iOS and Android plus one shell driver. Slicing would mean four review rounds covering decisions already settled. The single-PR cadence won the same way it did in Phase 8.

## 3. Break-it scenario

**The scenario PLAN.md / LEARNING_PROTOCOL.md L144 prescribes**: two real devices, both online. User A goes offline, deletes item X. User B online, edits item X's text. User A comes back online. What state does each device end up in, and is that what you'd expect?

This was not run as a fully scripted harness step but it's reasoned through here because the answer falls cleanly out of the engine's documented contract:

- **A's state at offline-delete**: local soft-delete on item X (`deletedAt = now`), one DELETE entry in the queue. UI hides the item.
- **B's state at edit**: PATCH on item X with `If-Match = X.updatedAt`. Server has X with `updatedAt = old_t` (whatever the last sync was). Match → 200, X.text becomes `B's text`, X.updatedAt becomes `t_b > old_t`.
- **A reconnects**: drainer wakes. The DELETE entry is at the head of the queue. Sends `DELETE /items/X`. Server: X exists, `deletedAt IS NULL` (B's PATCH didn't tombstone), `t_b` is the current `updatedAt`. The DELETE endpoint *doesn't* require `If-Match` (DELETE in this protocol is unconditional on the row's current value — see backend/docs/sync.md "Soft-delete (the contract)"). Server soft-deletes X. Returns 204.
- **A's drainer also reconciles items via `?since=`** as part of normal operation. Pulls the X row with `deletedAt = now_a` (the soft-delete A's DELETE just performed), drops the local row.
- **B's next reconcile pull**: pulls X with `deletedAt = now_a`, drops the local row. B's edit is gone — its text was committed to the server at `t_b`, then erased by A's later DELETE.

Is this expected? **Yes, given the protocol's contract**, but it's the most surprising LWW outcome to a user. B saw a successful edit (the PATCH returned 200), did nothing wrong, and lost their work. The semantic explanation is "DELETE is a write that wins by being later" — the DELETE arrived after the PATCH, so it wins under the engine's chronological reconcile order. It is *not* "the offline device's stale view of the world wins" — A's view at the time of the DELETE was that X was active, and the DELETE was a valid action on an active row. The undesired user-visible artifact is "B's edit silently disappears."

What would make this less surprising? The protocol's contract leaves a few options on the table for Phase 18 hardening:

- **Reject DELETE on a row that was edited after the local view's `updatedAt`.** Add `If-Match` to DELETE with the same semantics as PATCH. The drainer would then 409 on A's queued DELETE, the engine would reconcile (pulling B's edit), and the conflict would surface to the user (or be auto-resolved as "B's edit wins, A's delete becomes a no-op"). This is the strongest fix and matches HTTP/etag conventions for safe deletes. The cost is more 409 noise on real-world deletes that don't conflict.
- **Soft-delete with retention surface.** UI shows recently-deleted items with an undo affordance; B would see "X was deleted by A" rather than "X just disappeared." This is a UI fix, not a protocol fix, but it's the cheapest mitigation if the conflict is rare enough.
- **Treat DELETE-PATCH conflicts the same way as PATCH-PATCH conflicts** (the existing 409 path). Requires DELETE with `If-Match` (the same as the first option, framed differently).

The current Phase 9 contract is the simplest of these — DELETE is unconditional, conflicts resolve by timestamp, edits get silently lost when racing a delete. This is documented and acceptable for v1; Phase 18 is where we'd revisit if usage data shows real loss.

The break-it scenario's actual *implementation in the harness* would be: scenario (b) variant where A goes offline, deletes item X, while B simultaneously PATCHes X. The harness would assert both devices end up with no local row for X (because A's DELETE wins). I didn't add this as a scripted scenario in this phase — the four PLAN.md L386 scenarios are sufficient for "Done," and adding more scenarios is Phase 18 polish work. The reasoning above is the teach-back; the test would be one more `scenarioE_*` pair if/when we want it.
